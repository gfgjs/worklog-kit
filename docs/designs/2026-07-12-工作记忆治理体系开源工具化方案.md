## §1 背景与动机

**来源**。Scrollery 在长期开发中自研了一套两层记忆体系,已在本仓落地并经 CI 门禁验证:

- **易失层**:`/planning` skill(Scrollery 仓内 `.claude/skills/planning/SKILL.md`;本文引用的 Scrollery 路径均为**出处标注**而非活链接——本文自包含,关键契约已在 §7.1 正文化)——多阶段长任务在 `docs/planning/<日期>-<任务>/` 建 `task_plan`/`findings`/`progress` 三件套;零 hook、零 token 税、支持跨会话/跨机器续作。
- **永久层**:`docs/*` taxonomy + 两道门禁(`tools/check_docs.mjs` 断链/frontmatter/位置/收口契约,`tools/check_docs_index.mjs` 索引新鲜度)+ skill 分发器(`scripts/install-skills.mjs`)。
- **蒸馏缝**:收口时按 `closeout.md` 契约把易失层的耐久候选提升进永久层,由 CI 静态门机械校验(候选全覆盖/disposition 枚举/target 验存)。

**理论溯源**。两层结构对应 agent 记忆研究中的通识:上下文窗口=易失 RAM,文件系统=持久磁盘(Manus 式「文件即工作记忆」);易失→持久的**蒸馏**对应 CoALA / MemGPT / LangMem 里「短时工作记忆固化为长时语义记忆」的环节。市面工具多只做其一(纯 AI memory 框架无 CI 纪律;纯 docs linter 无 agent 自动喂料),两层咬合 + 静态门是本系统的稀缺点。

**为何开源化 + 隔离新仓**。①这套东西的价值(尤其 CI 强制的文档纪律 + closeout 蒸馏缝)对任何专业团队普适,不绑定 Scrollery 业务;②硬结构(门禁 + 结构)甚至不依赖 AI 也成立;③与 Scrollery 隔离可独立版本化/独立受众/独立治理,且避免把 Scrollery 私料混入公开物。

## §2 产品定位

**一句话**:一个把「工程文档纪律」做成 CI 可强制、并配套 AI working-memory 工作流的开发者工具——让文档从「靠自觉」变成「合并前置条件」。

**受众**:专业开发者与团队(用 AI agent 与否皆可受益)。

**两条消费路径(L1 双入口)**:

1. **全量治理**(治理层为主):`npx <tool> init` 在目标仓 stamp 出 docs 骨架 + 门禁 + CI + 生成式索引;团队获得机械强制的文档纪律,**不依赖任何 AI**。
2. **仅 skill**(易失层为主):`npx <tool> init --skill-only` 或作为 Claude Code plugin 装入 `/planning` working-memory 工作流;适合已有文档习惯、只想要 AI 长任务记忆的用户。**价值边界(R3-7)**:此模式只提供工作流指导,closeout 纪律回退为自觉、无机械强制;要机械强制必须装治理层——文档如实陈述,不暗示两路径等强。

**与既有品类的差异**:

| 既有品类 | 代表 | 缺口 | 本工具补什么 |
|---|---|---|---|
| ADR 工具 | adr-tools、log4brains | 只管 decisions,无长任务工作记忆,无跨类型 CI 门 | 全类型 taxonomy + 收口蒸馏缝 + 生成式索引 |
| Docs linter | markdownlint、Vale | 查文风/语法,不查「移动必修链/归档必横幅/收口必蒸馏」结构纪律 | 结构不变量 + 收口契约静态门 |
| AI memory 框架 | LangMem、MemGPT 类 | 运行时记忆,不落地为可 review 的仓内文件 + 无 CI | 文件化易失记忆 + 静态门 |
| Claude Code skill(裸) | 各类 skill | skill 本身只是行为规范,收口靶子须自备 | 把收口靶子(治理层)一并产品化 |

## §3 核心架构:两层记忆 + 蒸馏缝

沿用 Scrollery 已验证的骨架,**结构不变、承载形态按多人场景重塑**:

```
Layer 1  易失工作记忆(ephemeral)          Layer 2  永久知识库(durable, CI-gated)
─────────────────────────────────         ─────────────────────────────────────
planning/<unit>/                           designs/  reviews/  decisions/
  task_plan · findings · progress          runbooks/ worklogs/ archive/
  (一任务一目录,天然分片)                   + 索引按档:手工+不变量门 / 生成式 build(产物不入库)
        │                                          ▲
        │  closeout.md(蒸馏缝)                    │
        └──────── 逐候选提升 ────────────────────────┘
                  disposition ∈ {experience|decision|design|
                   runbook|completed|todo|code|test|no-promotion}
                  由 closeout 静态门机械校验
```

**保持不变**(已验证、跨场景通用的原语):

- 一任务一目录的易失三件套;无共享运行时锁/pointer/env(天然免协调)。
- closeout 收口契约(候选全覆盖不重不漏 / disposition 枚举 / target 验存或冻结引用 / verified=yes)。
- 「以文件为准,不信记忆」的接续纪律;archive/decisions 冻结不变量。
- 门禁把纪律机械化(团队场景价值更高)。

**按多人场景重塑**(详见 §4/§5):

- 索引与滚动状态从「单文件手工 + 不变量门」→「每单元一文件 + 可选生成式索引」(§4.1)。
- 工作线 ID 从「中心分配的字母」→「`lines/<slug>.md` 线实体文件」(§4.1,Q7 已裁):免中心分配,撞名在合并时成 git add/add 冲突原生暴露;文档另携稳定 `id` 与权威/生命周期不变量(§7.2,R2-C3)。
- 全中文内容 → 中文首发 + i18n-ready(§5)。
- 补齐软肋:任务名全局唯一、多人同任务归属、schema 变更权威、异构 agent(§4.3)。

## §4 多人协作重设计(第三支柱)

这是相对 Scrollery 单人版最大的 delta。Scrollery 版明确「用户基本在环、不做无人值守」,并行只想到「单人跨会话」而非「多人跨分支」。下面逐项把它升级到团队级。

### §4.1 每单元一文件 + 生成式索引(替代单文件手工 + 不变量门)

**问题回放**。Scrollery 版有两个共享单文件热点:`todo.md`(唯一滚动现状源)与 `README` 字母登记表。团队里每条 PR 都动 `todo.md` → 每次合并冲突;字母「下一空闲 = max+1」是**中心化分配**,两人同抢即撞号(索引门只能事后抓,已成瓶颈)。

**为何这里要反转 Phase 3 裁决**。Scrollery Phase 3 曾**拒绝生成式索引**,理由是「立项/权威文档/摘要等列是人工判断,frontmatter 里没有,硬做生成式要发明新字段且把人工蒸馏降级为机器倾倒」。该裁决在**单人**场景成立(单文件手工维护无冲突,生成式纯属多余复杂度)。但在**多人**场景取舍翻转:①单文件是冲突磁铁;②那条「元数据不在 frontmatter」的反对理由,其**解法恰好就是把元数据下沉进各单元 frontmatter**——而这同时把编辑面**分片**了。也就是说,消除反对理由的动作与消除合并冲突的动作是**同一个**。故多人版采生成式。

**设计**:

1. **每单元自带元数据**。每个 design/review/decision/runbook/worklog 的 frontmatter 承载索引所需字段,单元即真源:
   ```yaml
   ---
   id: 2026-07-12-gallery-dense-grid   # 稳定 ID:创建即得、改名不变(R2-C3);建议 <创建日>-<slug>
   status: active            # draft|active|snapshot|superseded|archived(机器面 ASCII,R3-1;状态机见 §7.2)
   type: design              # 见 taxonomy 枚举
   line: gallery-perf        # 引用 lines/<slug>.md 线实体(Q7 已裁,见 item2)
   title: Gallery dense-grid rendering
   authoritative: true       # 是否该线权威文档;唯一性不变量见 §7.2(R2-C3)
   created: 2026-07-12
   summary: One-line synopsis for the generated index.
   # supersedes: <旧文档 id>  # 取代关系成对声明(§7.2)
   ---
   ```
   「立项日/权威文档/摘要」这些**人工判断列**不再挤在一张中心表里,而是各单元在自己文件里各自申明——人工判断保留,但**零中心争用**。

   **字段表(P2 阶段 1 定案,2026-07-16;机器真源 = `src/lib/docmeta.mjs` 的 `DOC_FIELDS`)**。字段集属**元模型**(D-003):固定、由 `schemaVersion` 编号、变更须附迁移;具体枚举值(`status`/`type` 有哪几个)属实例,由配置声明。

   | 字段 | 必填 | 类型 | 适用范围 | 门禁校验 | 缺失时的梯子 |
   |---|---|---|---|---|---|
   | `id` | ✅ | token | 全部受检文档 | 非空 / NFC / 无空白与 `\|/\\` / **全局唯一** | `upgrade`(机器派生) |
   | `status` | ✅ | enum | 全部 | ∈ `config.status`;∈ `deprecatedStatuses` 则须移入 `archive/` | `baseline` |
   | `type` | ✅ | enum | 全部 | ∈ `config.types[].name` | `baseline` |
   | `line` | ✅ | text | 全部 | 非空(**阶段 2** 收紧为 `lines/<slug>.md` 实体引用) | `baseline`(人判) |
   | `created` | ✅ | date | 全部 | `YYYY-MM-DD` | `baseline` |
   | `title` | ⬜ | text | 全部 | 非空(缺省回落正文 H1) | — |
   | `summary` | ⬜ | text | 全部 | 非空 | — |
   | `authoritative` | ⬜ | bool | 仅 `canBeAuthoritative: true` 的 type | 字面量 `true`/`false`;资格由 types 元模型判 | — |
   | `authorityScope` | ⬜ | text | 仅 `authoritative: true` 的文档 | 非空;无 `authoritative: true` 时声明它 = 死配置 | — |
   | `supersedes` | ⬜ | idref | 全部 | 阶段 1 只验语法;**成对/存在性见 §7.2(阶段 3)** | — |
   | `supersededBy` | ⬜ | idref | 全部 | 同上 | — |
   | `owner` | ⬜ | text | 全部 | 非空(P3 多人用,§4.3) | — |

   **两把梯子(阶段 1 定案)**。新字段转必填即是新门,F-001 铁律「不得只上门不给梯子」适用;但梯子有**两把**,判据 = **机器能否派生该字段的值**:
   - **`upgrade`(数据布局迁移)**——值可由既有数据机械派生。`id` 属此:`<created>-<文件名 slug>`,`created` 本已必填、文件名本已存在,故 v2→v3 迁移**全自动补全**,人只复核。
   - **`baseline`(存量债立账,D-002/D-008)**——值是**人工判断**,机器派生不出来。`line` 属此:机器不知道一篇文档属于哪条工作线。这类缺口只能由人填,或由 `brownfield` 立账豁免存量、只对新文档 enforce。

   > **实测靶场代价(2026-07-16,`/d/workspace/scrollery` @ `dev` `13a3192`,只读)**:165 篇 md 中 **48 篇三件套 + 32 篇 archive 本就豁免**,真正受检 **85 篇**;其中缺 `line` **10 篇**、缺 `created` **0 篇**、缺 frontmatter **0 篇**;archive 里带 frontmatter 的仅 **4 篇**。两条推论:①`created` 零缺失 ⇒ **`id` 派生对全部 85 篇可行**,`upgrade` 这把梯子不会中途退化成「一半自动一半人工」;②`line` 缺口只有 10 篇,`baseline` 立账后即清零,阶段 1 对靶场**不构成迁移悬崖**。
   > **⚠️ 口径更正**:§4.1 item2 ④ 记的「124 文档有 `line:`」是**另一种口径**(含 archive 与三件套,量的是「全 docs 树里出现过 line 字段的文件数」),与此处的 governed=85 **不可比**。旧数不是错的,是量的不是同一个东西——两处并存时勿相减,`165 − 124 = 41` 这种算法会得出一个从未被测量过的数(施工时确曾如此写下,已更正)。

   > **为什么 `summary` 不设必填**(阶段 1 定案,推翻「索引要它就必填」的直觉)。门对 `summary` 只断言得了**非空**,断言不了**有用**。于是「必填」在存量仓里只会产生一种结果:165 个满足门禁而不告诉任何人任何事的占位符——**门亲手制造出它本该防的东西**。这正是 Phase 3 当年拒绝生成式索引的原话「把人工蒸馏降级为机器倾倒」,与 D-005「空读数与好读数在表上同形」同构。故 `summary` 可选;生成式索引里缺它就是**空单元格**,那个可见的空白比一个写着「TODO」的绿灯更有压力。`title` 同理但另有一层:它与正文 H1 重复,必填即造出一对**必然漂移**的双写——R2-C1 已把 drift 语义从全文清剿过一次,不在此处重新引入。故 `title` 可选、缺省回落 H1,**且门不校验二者是否一致**(那正是 drift gate)。

   **`id` 派生规则(v2→v3 迁移用;仅一次性播种,之后冻结)**:①取文件名(去 `.md`)→ slugify(D-007 同一套);②不以 `YYYY-MM-DD-` 开头则前缀 `created-`;③与他人撞号则再前缀父目录名;④仍撞则**报告并拒绝迁移**,交人裁。为什么撞号真会发生而非理论风险:本仓 `docs/README.md` 与 `docs/worklogs/README.md` 同名同 `created`,②之后二者均为 `2026-07-16-README`——**迁移命令在自己的 dogfood 仓里第一步就会撞**。

   > **`id` 与 `line` 的分工**(消解「两个都像唯一键」的直觉):`id` 是**文档**的主键(全局唯一、改名不变、supersedes 挂它);`line` 是**归属**(同线多文档共用同一个值是正常复用,见 item2 ③)。二者不竞争。

2. **工作线 = `lines/<slug>.md` 线实体文件(Q7 已裁,2026-07-13)**。开线 = 新建 `lines/<slug>.md`(一句话使命 + 可选 owner),无需向任何中心表登记取号 → **消除分配瓶颈**;文档 frontmatter 的 `line` 字段只**引用**该 slug。唯一性语义随之可判定(消解 R2-C2 的自相矛盾):①线的唯一声明 = `lines/` 内文件名,两人撞名 = git **add/add 冲突**,合并时即被原生暴露(早于任何事后门);②门禁校验「文档引用的 line 必须存在对应实体文件」;③同线多文档共用 `line` 值是**正常复用**,不再被误判为碰撞——文档级唯一性由 `id` 承担。字母表连同「下一空闲字母」不变量一并退役。(M-5 的「可检出而非根除」结论保留:实体文件把检出左移到合并时刻。)

   **④ brownfield 迁移悬崖与迁移命令(F-001;2026-07-16 裁定,同日按 Scrollery `dev` 实测数据修正)**。

   > **数据形态以实测为准(2026-07-16,`/d/workspace/scrollery` @ `dev` `cd0443c`;值样例已按 R4-18 于 fresh-export 合成化,形态与计数保真)**。本块初稿假设「存量 `line` 值就是字母,迁移=人工给字母起名」——**实测证伪**。真实形态是**字母与 line 两套并行、且已互相脱钩**:
   > - **字母登记表**(`docs/README.md` §「工作线字母登记表」,19 行 **A–S**):字母 = **`todo.md` 分节号**,表头「新立项**先来此取号**」——取号规则白纸黑字存在。
   > - **`line:` frontmatter**(124 文档,**54 个 distinct 值**):值是**语义名**(`legacy_2020`、`AI 标注流水线深审`、`存量治理`),**不是字母**。
   > - **混合形态 12 个**:`巡检修复(F)`、`丙线完善(K)`、`parser 模块拆分(T)`——语义名尾挂字母。
   > - **三处 drift 实证**:①54 个 line 值 vs 19 个字母 → **35 个工作线从未取号**(中心表被静默弃用,非被反对);②`(T)` 已在野但**不在登记表**(表止于 S);③登记表 O 行原注「**原误取 I 撞号,2026-07-10 改 O**」→ **中心分配的撞号是已发生事件,不是理论风险**。

   P1 MVP 里 `line` 是**零校验自由文本**(schema 无此字段;`check-docs` frontmatter 只校验 `status`/`type`),故上述三种形态**混存不报错**——但这是零校验的**副产物,不是设计**。本 item 的引用门一旦上线,**54 个 line 值全部同日变违规**(无一有对应实体文件)。故**引用门与迁移命令须同批交付,不得只上门不给梯子**。

   **迁移命令实际须做四件**(按实测形态,非初稿的「扫 line 值起名」):
   1. **解析 + 剥离**:读 54 个 distinct `line` 值 → 剥 `(X)` 字母尾 → slugify(规则见 item2 ⑤:剔文件系统非法字符 + 空格→`-` + NFC 归一;**中文原样保留**)→ 生成 `lines/<slug>.md` 占位实体。**语义已在值里,slug 可自动派生**——人工只需**复核**,不需**发明**(初稿「机器不知道 K 指什么」的判断对**纯字母**成立,对**实测形态不成立**)。
   2. **登记表归并**:19 行字母表按工作线名匹配 line 值 → 合并进对应实体的 frontmatter(保留「立项日/权威文档」两列的人工判断,§4.1 item1)。
   3. **报告三类失配**(机器判不了,必须人看):①登记表有字母但无任何 line 值引用(死号);②line 值带 `(X)` 而 X 不在表(如 `(T)`,野号);③35 个无字母无登记的 line 值(表外线)。
   4. **`todo.md` 分节号改写**:`## A. 发布流水线` → slug 化。**字母同时是分节号,故退役字母 = 改滚动状态源的结构**,不只是删一张表。

   这把 R3-3「档间切换是数据布局迁移,不是配置开关」从**索引档位**扩到**工作线命名空间**——同一原则,第二个适用面。
   > **取号触发源澄清**:工具自身**零取号**。旧项目的字母取号来自其**自身**的两件东西——README 字母登记表(数据)与旧 skill 文本「取下一空闲字母」(行为);`install-skills` 覆盖后者即断掉唯一执行者,字母表随之退化为无人执行的死数据。故「装本工具会触发取号」是**伪命题**;真问题只是存量字母的迁移时机(建议:P2 引用门上线前不动,与迁移命令同批处理)。

   **⑤ slug 字符集与文件名语言(R4-15;2026-07-16 已裁)**。**中文(及一般 Unicode)文件名是一等公民**——`lines/<slug>.md`、planning 任务目录、docs 文件名的语言随使用者内容,这是首要消费场景(用户明示第一需求)的硬要求;R3-1「机器面锁 ASCII」的适用范围随之**修正**(见 §5):文件名**划出机器面、归使用者内容面**。落地四则:
   - **slug 派生规则**(迁移命令与 skill 同一套,不发明第二种):剥 `(X)` 字母尾 → 剔除文件系统非法字符 `\ / : * ? " < > |` → 空格→`-` → **NFC 归一**。与 SKILL.md 任务目录名规则同源;中文原样保留,故 F-001「slug 自动派生、人工只复核」对中文值**继续成立**。
   - **门禁 NFC 兜底**:引用门对 `lines/`、`status/` 文件名断言 **NFC 编码**(非 NFC 即红,提示重命名);`line` 引用解析**先 NFC 归一再比对**——把 macOS(NFD)/Windows·Linux(NFC)的归一分叉挡在入库前,Q7「撞名 = add/add 冲突」的卖点在归一后恢复有效。**归一必须在比对之前**:否则一个 NFD 的 target 会先撞「你指错线了」,而用户指的线其实没错、错的是编码——报错把他送去改线名,他怎么改都改不对(2026-07-16 施工时实测,已入 selftest)。
   - **⚠️ 事实更正(2026-07-16 施工时实测,推翻本条原措辞)**:上一条原写「同一个**中文名**在两台机器上会是两个不同的字节串」——**对纯中文不成立**。NFD 只分解有正则分解映射的字符;**CJK 统一表意文字没有分解映射**,天然 NFC 稳定:`'中文工作线'.normalize('NFD') === '中文工作线'` 为真(`'日本語'` 同)。真正会分裂的是**谚文**(`한` 1→3 码点)、**带浊点/半浊点假名**(`が` 1→2)、**带变音符拉丁**(`é` 1→2)。推论有三:①**NFC 门仍要留**——混排项目里的假名/谚文/重音拉丁是真会分裂的,故本条不因此作废;②但它**保护的不是中文**,原理由属想当然;③**「中文一等公民」的残余风险比原先记的小得多**——下一条「真失灵再修」所指的失灵,在纯中文场景根本不存在。教训:拿纯中文测 NFC 门等于测一个恒真命题,selftest 会全绿而什么都没证;fixture 已改用假名与谚文。
   - **排序确定性**:生成器排序键改述为「`created` + slug 的 **UTF-8 码点序**比较」——含中文时码点序依然确定,M-10 的实质(不依赖 locale collation)保留,仅「纯 ASCII」措辞作废(§7.2 已同步)。
   - **残余风险已知接受**:极端平台/文件系统组合下(如未开 `core.precomposeunicode` 的旧 mac 环境 + **可分解字符**的文件名)add/add 冲突检测仍可能不触发——**真失灵再修**(用户 2026-07-16 原话),不为未发生的失灵预支复杂度。按上条更正,该风险**不覆盖纯中文文件名**。

   **⑥ `lines/` 与 `status/` 实体 schema(P2 阶段 1 定案,2026-07-16)**。二者都是**手写入库文档**,不是生成物:`lines/` 是线的身份声明,`status/` 是收口的滚动状态靶点(D-014/D-016,每线恰好一文件);生成物只有 `INDEX.md`/`STATUS.md`,不入库(C-3)。**故二者不需要第二套 schema**——它们就是普通受治理文档,走 item1 的同一张字段表,各占一个 `type` 而已:

   | 实体 | `type` | `line` 字段 | `authoritative` | 归档形态 | 落地阶段 |
   |---|---|---|---|---|---|
   | `lines/<slug>.md` | `line` | 指向自身 slug | 恒 false(线不是自己的权威文档) | 移入 `archive/` + 横幅指去向;**归档线不得再被 active 文档引用**(§7.2) | 阶段 2 |
   | `status/<slug>.md` | `rolling-status` | 所属线 | 恒 false(滚动状态是时点记录,不是某条线的当前答案) | 随线归档 | 阶段 4 |

   两个 type 均 `canBeAuthoritative: false`(判据同 §7.4 内置表:「它是不是某条线在某范围内的**当前答案**」)。**加这两个 type 是实例变更、不触发升版**(D-003),故与各自落地阶段同批进 DEFAULTS,**不在阶段 1 预先塞入**——那会造出一个当下无人读的 type,与 §7.4 拒绝「死配置」的判据自相矛盾。

3. **滚动状态分片**。`todo.md` 单文件 → `status/<slug>.md` **每工作线恰好一文件**(R5-C6;2026-07-16 已裁 → D-016,原「或每任务一文件」**作废**),两人改不同线永不冲突;「现在做到哪」由生成器从各 `status/*` + 任务 frontmatter 聚合成只读的 `STATUS.md`。**粒度取「每线」而非「每任务」的三条理由**:①冲突隔离的单位就是线——同线两人本就要谈,跨线两人不该撞,每任务分片只是把同一条线的状态再切碎,不多隔离一分;②滚动状态的读者问的是「这条线现在到哪」,每任务一文件等于把答案摊到 N 个文件、又得靠生成器拼回来;③`line-status` disposition(§7.4)要从归档任务的 `line` **求解唯一靶点**——每任务一文件则同线多任务给出多个候选靶点,靶点不再唯一,固定靶点强度随之失效。文件名 slug 规则同 item2 ⑤(中文一等公民 + NFC)。

4. **生成式索引:`index build` + `check`,产物不入库(C-3 已裁;R2-C1 语义统一)**。`index build` 扫描各单元 frontmatter 产出人可读 `INDEX.md`/`STATUS.md`;产物列入 `.gitignore` **不入库** → 两分支根本不含派生文件,**真零合并冲突**(这是生成式相对不变量门的唯一实质好处,唯 gitignore 才兑现)。**CLI 面已裁(R4-17,2026-07-16)**:`index build` / `index check` 为显式子命令;**裸 `worklog index` = 按档别名**(不变量门档→不变量门校验,生成式档→`index check`)且执行时打印所指——现役 npm script/CI 模板/README 已把裸 `index` 固化为门禁,别名保兼容,打印防 R2-C1 式「两套 check」在 CLI 层重演。**索引是派生物,从不手工编辑、从不提交**;因此本工具**没有 drift gate**——「生成物≠源即红」「忘记重生成」「索引不新鲜」这些概念随产物不入库整体消解,pre-commit hook 亦无物可拦、不做。CI 侧跑 `check`:断言**生成成功 + 源结构一致性**(line 引用存在 / 权威唯一 / worklogs 双向 / frontmatter 齐备,见 §7.2)。
   - **可见性补偿(R3-2)**:仓内/GitHub 看不到索引是 gitignore 的真实代价,补偿分层承诺——本地预览 `npm run docs:index` 是**默认路径**(README 如实陈述);CI 发布的最低承诺 = **Actions artifact**(零新增基建);docs site(如 Pages)属 opt-in 增量,不进 MVP;发布管线细节在 P2 上生成式档前裁定。

**冲突面对照**:

| 编辑对象 | Scrollery 单文件版 | 本多人版 |
|---|---|---|
| 滚动状态 | 人人改 `todo.md` → 每 PR 冲突 | `status/<slug>.md` 分片(每线一文件,D-016)→ 仅同线冲突 |
| 工作线登记 | 中心字母表 + max+1 分配 → 撞号 | `lines/<slug>.md` 线实体 → 撞名 = add/add 冲突合并时原生暴露 + 门禁引用检查兜底 |
| 人可读索引 | 手工维护 + 不变量门 | 派生生成,产物不入库(C-3)→ 真零冲突 |
| 单元元数据 | 集中在中心表 | 各单元 frontmatter 自持 → 仅改同一单元才冲突 |

**诚实代价**:生成式要维护一个生成器(更多代码),且索引仓内不可直接浏览(补偿分层见 item4)。**不采「CI 自动 commit 生成物」**(自动 commit 有其自身危险:污染作者归属、与保护分支/签名冲突)——产物不入库后,此选项连同「忘记重生成」问题一并失去存在前提。

**索引形态按团队规模可选(M-4)**。生成式并非唯一形态,而是**最重的一档**;强加给 2–5 人小团队可能杀鸡用牛刀(并发触碰 docs 的 PR 本就少)。故工具提供**三档梯度**,由 `init` 按团队规模选、或 `.<tool>rc` 配置:

| 档 | 机制 | 适用 | 冲突面 |
|---|---|---|---|
| **不变量门**(Scrollery Phase 3 形态) | 手工维护索引 + 结构不变量静态校验(目录↔登记双向一致等;与已退役的生成物 drift gate 无关) | 单人 / 极小团队 | 索引单文件会冲突,但零生成器负担 |
| **append-only 登记** | 登记表只增不改(新行追加末尾)+ 唯一性检查 | 中小团队 | 冲突减少但非免疫:两分支同时追加文末仍相邻行冲突(R2-m1);高并发可改每登记一文件 |
| **生成式**(本节主设计) | 各单元 frontmatter → `index build` 产出索引(不入库)+ `check` 结构校验 | 中大团队 / 高并发 | 真零冲突(C-3 gitignore) |

MVP 首发仅需**不变量门**档(最简、已在 Scrollery 验证);append-only 与生成式作为 opt-in 增量,待 dogfood 验证需求后再上(呼应 L8 MVP-first)。

**档间切换是数据布局迁移,不是配置开关(R3-3)**:不变量门档(中心登记表 + 单文件滚动状态)与生成式档(元数据下沉各单元 frontmatter + `status/` 分片 + `lines/` 实体)的数据承载不同,升档须**一次性迁移**——工具提供迁移命令,或至少文档化手工步骤;且 skill 的收口靶点(登记行 / todo 回写)随档位变,skill 文本须按档参数化(读配置定收口靶点,§7.3/§8.1)。

### §4.2 语义合并竞态消解

全局不变量在多人下的经典陷阱:PR-A、PR-B 各自绿,合并后不变量破了(git 文本合并干净但语义破)——即 semantic merge conflict。

- **索引类**:本版索引为派生物且不入库,合并后按需重建即恢复,无残留破态。
- **分配类**:线实体文件免中心计数器 → 无分配瓶颈;撞名 = add/add 冲突在**合并时**原生暴露(M-5 的「事后可检出」升级为「合并时可见」)。
- **主干完整性(R2-C6 措辞修正)**:main push 后再跑全部结构门是**事后探测 + 恢复兜底**——它**拦不住**已进主干的破态,正文不得称「拦下」。多人档的**要件**(非建议):required status checks + up-to-date-before-merge,或 merge queue(在 `merge_group` 事件上对**拟合并结果**跑门)——这才是把 semantic merge conflict 挡在主干之外的机制;单人/极小团队档可只留事后探测。

### §4.3 软肋修复(团队下必补)

| 软肋 | 症状 | 修法 |

**同任务多人写模型(R2-C7,已裁①混合,2026-07-13)**:

- **单人档(默认;Scrollery 验证形态)**:常规三件套,单文件 `progress.md`——零新增成本,MVP 即此档。
- **team 模式(多人同任务)**:任务创建时在 `task_plan` frontmatter 声明 `mode: team` + `owner`/`collaborators`;`progress` 改**每事件一文件** `progress/events/<UTC时间戳>-<author>-<seq>.md`(ISO 8601 紧凑时间戳,文件名字典序即时间序;同秒撞时靠 `<seq>` 去歧义),人可读时间线视图由 `index build` 聚合、不入库。`task_plan`/`findings` 保持共享单文件(低频编辑,罕见冲突值得人工审);**候选 ID 带作者命名空间**(`F-<author>-NNN`/`D-<author>-NNN`),消 ID 分配竞态。
- **模式在任务创建时声明;中途切换 = 一次性迁移**(`progress.md` → `events/`,与 §4.1 档间迁移同款语义),不做隐式混用。
- **closeout 秩序**:仅 `owner` 发起 closeout;closeout PR 开启后候选账**冻结**,迟到候选进下一任务/下一轮,不回头追加。
- 该模型与三档索引梯度**同构**(按团队规模选形态、单人档零成本、升档=显式迁移),共用同一套「档位」心智,不新增第二种配置哲学。
- **R1 并发基线读数(2026-07-17,验证场 D-026 实测,本块设计的直接实证输入)**:三人同基并发下**共享三件套整体是冲突面**——dev-B 并入 progress/findings/task_plan 各 1 处内容冲突 + 撞名 4 add/add(7/7 文件),不止 progress 文末;会话块**无作者字段**,溯源全靠 `(dev-X)` 后缀惯例(F-004,双 agent 互不知情独立报出)——events 档(上方①)与作者命名空间由此从推断变实测。任务名唯一在对照组消掉整类 add/add 冲突(dev-C 独名净并)。台账:`docs/worklogs/2026-07-17-P3验证场与并发基线/`。
- **施工落地(2026-07-17,P3 施工)**:上述①混合全量落地——`mode: team` + owner/collaborators 声明门(E1);events 档文法门(E2~E4:定宽 UTC 时间戳-作者-两位 seq,字典序即时间序;事件**零 frontmatter**,文件名即作者机器真源,D-027);候选 ID 作者命名空间门(E5)+ 仅 owner 收口门(E6);`worklog team`(solo→team 一次性迁移,含存量候选 ID 重命名梯)与 `worklog closeout`(收口机械步收敛,agent 环境下工具权限提示即「用户批准收口」按钮)两条命令(D-029);timeline/STATUS「在施任务」视图由 `index build` 聚合、不入库。schemaVersion v5 不动(D-028)。实现契约见 `docs/designs/2026-07-17-team写模型events档与closeout命令.md`(authoritative,上位裁决仍在本块);台账:`docs/worklogs/2026-07-17-P3多人软肋施工/`。

**closeout 适用边界(M-7)**。收口契约(F-NNN/D-NNN 候选账 + 蒸馏 + `closeout.md`)对顺手修 bug 的 drive-by 贡献者太重,不应强加到**每个** merged PR。边界:**closeout 只作用于「长任务 / 工作线级」**——即建了 `planning/` 三件套的任务;PR 级小改(无三件套)豁免。触发判据即「该任务是否存在三件套」(门禁对含三件套的归档目录强制 closeout,普通 PR 不涉及)。「够长要建三件套」的判定沿用 skill 既有阈值(≥5 步 / 跨会话),由发起者 / 维护者定,门禁不猜。

## §5 语言与 i18n:中文首发,i18n-ready

**原则:工具产出语言 ⟂ 使用者项目内容语言**。工具只 i18n **自身** UI/报错,绝不翻译使用者的文档内容;使用者可把自己项目的文档写成任何语言。

**第二刀:机器面 ⟂ 展示面(R3-1;范围经 R4-15 修正,2026-07-16)**。第一刀切的是「工具自身 ⟂ 使用者内容」,还须切第二刀:frontmatter **字段名与枚举值**(status/type/disposition)、closeout 表头、CLI 子命令等**机器可判定面锁 ASCII canonical**(如 `status: active`);**人可读面**(报错文案、生成索引的表头、模板 prose)走 catalog 本地化——中文首发即「ASCII 枚举 + 中文展示」。**文件名不在机器面**(R4-15 修正:原文曾把「文件名约定」列入 ASCII 锁,与 F-001「slug 自动派生」在中文 line 值上互斥;已裁**文件名归使用者内容面**,中文一等公民,门禁以 NFC 归一兜底,细则见 §4.1 item2 ⑤)。这消解一处潜在自相矛盾:L5 的「模板中文首发」指模板 **prose** 中文,**不指枚举值中文**;若中文模板配中文枚举值,未来国际化 = schema breaking change(major),catalog 映射层从第一天就拆掉这枚雷。(Scrollery 现役中文枚举是单仓自用的合法选择,不随抽取继承。)

> **R3-1 的适用主语 = 本工具自身,不是消费者实例(2026-07-16 靶场实测后澄清)**。「机器面锁 ASCII」若被读成「**你的** status 值必须 ASCII」,首要消费者 Scrollery 装本工具的第一步就成了「迁移 165 个文档的 frontmatter」——**那是误读,且代价差了一个数量级**。`status`/`types` 是**实例**,D-003 已裁实例可配;R3-1 约束的是**本工具的默认值、schema 字段名与 disposition 元模型**。实测(`dev` `d7200f1` 只读副本):照抄其中文 `STATUS_ENUM` 与 19 个中文 type 值配置 → `loadConfig` errors=0,两门全绿,**零文档迁移**。故 ASCII canonical 对消费者是**建议**(schema `description` 措辞即「建议 ASCII」),不是门;工具不替使用者的 taxonomy 做语言决定——那与本节开篇「工具产出语言 ⟂ 使用者项目内容语言」是同一条原则的第二个适用面。

- **MVP 首发纯中文(C-2/M-8)**:门禁报错、`init` 输出、模板骨架、`SKILL.md`、README 首发中文——这是作者母语与首个 dogfood 场景,最快验证。原 v0.1「英文默认」在验证前反转:**先证明它有用,再谈国际化**(呼应 L8 MVP-first)。
- **i18n-ready 架构,但不做多语翻译**:即便首发单语,所有面向用户的字符串**从第一天就外置**(如 `locales/zh.json`),门禁/CLI 文案走 catalog 查表,语言经 `--lang` / `.<tool>rc` / `LANG` 选择。如此加英文/他语只是新增一份 catalog、零改逻辑。**英文 catalog 推迟到有采纳信号后再补**(m-8:专业开发者读单语报错无碍,过早双语是纯维护负担)。**MVP 的 fallback 链即 zh**(唯一 catalog;R2-m2 已裁纯中文)——`requested → en → …` 之类的链待英文 catalog 上线后再定义,不预设不存在的落点。
- **术语一致性**:提供 glossary(working memory / closeout / distillation / disposition / work-line / drift gate 等),中文文案保留英文术语(与 Scrollery 约定一致),避免翻译漂移。
- **转公开前评估英文门面**:面向国际贡献者的最小英文集(README/CONTRIBUTING)在 P5 转公开前按需补(见 §12),非 MVP 阻塞项。

**分发形态(🔵 OPEN,§13)**:倾向**兼有**——

- **npm 包 + `npx <tool> init`**:主分发路径,`init`/`index`/`check` 皆为子命令;版本化清晰,升级方便。
- **template repo**:GitHub「Use this template」一键起,内含已 wired 的 CI 与 docs 骨架;适合不想过 npm 的用户。
- 二者共享同一 `src/`,template repo 由 CI 从主包生成,避免双份漂移。

**双入口落地(L1)**:`init`(全量)与 `init --skill-only`(仅装 skill)同一命令两参;Claude Code plugin 作为**可选薄封装**指回本仓(锦上添花,非地基)。

**安装拓扑与升级模型(R2-M3)**。唯一安装拓扑 = **thin runner**:引擎(门禁/生成器/分发器)驻 npm 包内,用户仓只落**配置 + templates 实例 + ci.yml**,不复制引擎脚本。
> **边界澄清(R4-02;2026-07-16 施工时坐实)**:「引擎驻包」**不等于**「文档也驻包」。**人和 AI 要读的东西必须在消费仓里**——模板、收口手册、skill、版本 manifest 全部 stamp 进去。此前 SKILL.md 与手册引的是**包内** `templates/`,而消费仓根本没有那个目录:那不是一条次优的路径,是一条**走不通**的指路。落地:`init` stamp `.worklog/templates/{task_plan,findings,progress,closeout}.md` + `docs/runbooks/closeout.md` + `.claude/skills/planning/SKILL.md` + `.worklog/manifest.json`,skill/手册一律引仓内路径。**本仓 dogfood 测不到这条**(包==仓,`templates/` 在这里永远存在)——故由 e2e 在 temp 消费仓里守(§10)——杜绝「一次性 template generator」退化,semver 升级即 `npm update`。stamped 文件与配置携带 `schemaVersion` + 生成时工具版本(manifest);`init` 支持 `--dry-run` 与幂等重跑(已存在文件默认不覆盖,报冲突清单);`doctor` 诊断用户仓与当前工具版本的兼容性(含 stale trio 报告,R3-9);`upgrade --dry-run` 预览迁移,migration registry 按 `schemaVersion` 逐级迁移(含 §4.1 档位升档、§4.3 solo→team 迁移);所有写入原子(tmp→rename)+ 先备份可回滚。break schema/CLI 契约 = major + 附迁移条目,semver 由此对用户仓成为**可升级契约**而非空话。

**消费仓版本钉(R5-M3;2026-07-16 已裁 → D-017)**。thin-runner 的代价是**引擎不在消费仓里**,于是「跑的是哪个版本的引擎」成了消费仓自己答不出的问题——而门禁的判定结果**依赖引擎版本**,答不出即不可复现。落地三则:

- **`init` stamp 版本 manifest**(`.worklog/manifest.json`,入库):记录 stamp 时的 `worklog-kit` 版本、`schemaVersion`、profile、`docsDir`、stamp 日期。`doctor` 据此比对当前引擎版本,给出兼容/需 upgrade 判定;`upgrade` 据此定迁移起点。
  > **落点(2026-07-16 施工时定)**:manifest 与消费仓可达模板同住仓根 `.worklog/`,不再各自散落 dotfile;与 R5-M7 建议的生成物目录 `.worklog/generated/` 同一个家。**放仓根而非 `docsDir/` 下**是硬约束:`docsDir` 内每个 `.md` 都要过 frontmatter 门,而模板正文带 `<任务名>` 占位符必然不合法——放进去等于 `init` 完当场红。`.worklog` 同时进 `sourceExclude` 默认(理由同 `.claude`/`.github`:模板里的 target 样例是**示范**不是引用,纳入 1b 会误报)。
- **CI 调用钉到精确包版本**,不用裸 `npx worklog`:裸 `npx` 在无本地 bin 时会去 registry 抓**最新**版,既不可复现,也不能证明解析到的就是 `worklog-kit` 的 `worklog`(同名 bin 冲突不报错,静默跑别人的)。模板改为 `npx --yes worklog-kit@<精确版本> worklog ...`;若消费仓本就是 Node 项目,可改钉 `devDependency` + `npm ci` 后跑本地 bin。
- **不假设消费者是 Node 项目**:`init` **不写** `package.json`、不改消费仓依赖拓扑——Scrollery(Rust/Tauri)与任何非 Node 仓都必须能装。故 manifest 是独立文件而非 `package.json` 字段,CI 走 `npx --yes <pkg>@<ver>` 而非 `npm ci`。

**押注治理层、skill 当可替换插件(m-11)**。若 Anthropic 日后把 skill/memory 一方标准化(plugin marketplace / 官方 skills),skill 半边可能被蒸掉;故产品**主价值押在不依赖 AI 的治理层**(门禁 + 结构 + closeout 蒸馏缝),skill 视为可被官方或他家替换的一等**插件**而非地基——这也是 L1「两层平级」在风险维度的兜底。

**dogfood**:本仓自身用这套治理管理自己的 `docs/`——最强的活文档 + 回归防线。

## §7 门禁套件重设计(通用化 + i18n-ready + 可配置)

从 Scrollery 三脚本抽取,已勘察耦合面很小,主要工作是**通用化 + 英文化 + 抽配置**:

### §7.1 check-docs(断链 / frontmatter / 位置 / 收口)

- **`walk()` 加存在性守卫**:Scrollery 版对 `src/`/`scripts/` 等根目录不设守卫,新项目缺 `src/` 会 `readdirSync` 抛 ENOENT。加一行 `if (!existsSync(dir)) return` 根治(顺带加固,零行为变化)。
- **1b 代码引用扫描去硬编码**:Scrollery 版写死 `src-tauri/`、`Cargo.toml`、`copy.bara.sky`。改为**探测顶层源码目录 + 可配排除表**——⚠️ 纯自动探测会误扫 `dist/`/`vendor/`/`.venv/` 等大生成树(慢 + 对生成物里的 docs 路径误报),且排除表本身项目相关。故诚实降级为「默认排除常见构建/依赖目录 + `.<tool>rc.sourceRoots` 可覆盖」(M-9;非 v0.1 声称的「纯自动」)。去掉 `plan-docs` 旧前缀兼容(Scrollery 专属历史)。
- **枚举外置**:`STATUS_ENUM`、disposition 集合、type 枚举从代码常量提到**配置文件**(§7.4),门禁读配置,不写死特定 taxonomy。
- **报错走 i18n catalog**:所有 `report(loc, msg)` 文案外置查表,首发中文 catalog(L5;英文后补)。
- **收口契约正文化(R2-C4)**:不再以「原样保留 Scrollery 范式」代替定义——源契约散在 Scrollery 私仓三处(worklogs README + skill 模板 + check_docs 收口节),公开受众永不可达,自包含是硬需求。正式 schema 如下,**本节即唯一真源**:
  - **候选声明源**:三件套两张表(findings「耐久提升候选」表、task_plan「关键决策」表)的「候选 ID」列;叙事提及不算。ID 形态 `F-NNN`/`D-NNN`(team 模式带作者命名空间 `F-<author>-NNN`,§4.3)。
  - **closeout 处置表列**:候选 ID / disposition / target / locator / 去重证据 / N-A 理由 / verified。
  - **全覆盖不变量**:声明候选集合与处置表 ID 集合恰好相等(不重、不漏、无未知 ID)。
  - **cardinality(R5-C2;2026-07-16 已裁 → D-012)**:**一候选恰好一处置行**,ID 即行主键;「全覆盖」= 声明 ID 集合与处置 ID 集合**逐元素相等**。一个发现若确实要产出多个落点(既进 design 又补 test),**拆为多个原子候选**并在摘要互链,不用同 ID 多行。原措辞「同 ID 多行、以(ID, disposition, target)组合唯一」**作废**——它与实现(`check-docs.mjs` 的 `seen` 按 ID 去重)、skill/template/README(三处均写「每候选恰好一行」)分叉成三套契约两个答案;裁向多数侧,使覆盖判定与去重判定共用同一把钥匙(ID),元模型不必为罕见形态引入组合键。
  - **per-disposition 语义**(与 §7.4 结构化配置一一对应):docs 类(experience/decision/design/runbook/completed)target = `repo:<仓根相对路径>`,门禁**验存**;`todo` target 按配置的 targetKind 分档——不变量门档 `fixed`(固定为配置声明的 `repo:docs/todo.md`),生成式档 `line-status`(配置只声明根目录,靶点由归档任务的 `line` 求解,见 §7.4);`code`/`test` target = **冻结引用** `repo:<路径>@<commit 短hash>`(只验 grammar 不验存在,永不回改);`no-promotion` target = `—` 且 **N-A 理由必填**。
  - **仓根 containment(R5-C1)**:一切 `repo:` 路径**必须解析后仍在仓根内**——拒绝绝对路径、`..` 上跳、空段;`repo:../<仓外文件>` 一律非法,不得因该文件在磁盘上存在就验存通过。**验存的语义是「落点在本仓、随本仓演进」,不是「磁盘上有这个文件」**。
  - **固定列 schema(R5-C1)**:处置表表头列名、列数、顺序**均受校验**,不止首列含「候选 ID」即放行——按列位置解构的实现,列漂移会静默错位取值。
  - **verified 无条件校验(R5-C1)**:`verified` 是**全 disposition 共用**的收口断言,`no-promotion` 不例外(「不提升」也须有人确认这个决定已做完)。实现上它必须在 per-kind 分派**之外**判定,不得被任何分支的提前 `continue` 跳过。
  - **去重证据列**:`new`(新落点)或 `repo:<路径>`(复用既有知识);no-promotion 填 `—`。
  - **verified 语义**:只接受 `yes`,含义 = 提升动作已在**同一变更集**内完成且落点可见——closeout、目录归档、永久文档写入须同 commit/PR **原子完成**(同变更集新建的 target 在合并时即存在,验存无鸡蛋问题)。
  - **冻结边界**:归档后三件套正文冻结;closeout 语义列(ID/disposition/理由)冻结;仅 docs 侧 target 因目标文件迁移可在**同一移动 commit** 更新(target 消失即门红,机械强迫修链);冻结引用永不回改。
  - **最小样例**:合法 `| F-001 | experience | repo:docs/experience.md | §N 标题 | new | — | yes |`;非法 `| F-002 | code | repo:src/foo.mjs | — | new | — | yes |`(code 缺 `@<commit>` 冻结引用)——selftest 正负例即由此对生成(§10)。

### §7.2 build-index(生成 + 结构校验;取代 check_docs_index 的纯不变量门)

- 形态从「校验手工索引不变量」→「**`index build` 从各单元 frontmatter 生成索引(产物不入库)+ `check` 校验源结构**」(§4.1;R2-C1 清剿后,全文不再有任何 drift 语义)。
- 生成:扫 `designs/reviews/decisions/runbooks/worklogs` 各单元 frontmatter,按 `line`/`type`/`authoritative` 聚合,产出 `INDEX.md` 与 `STATUS.md`(带 generated 护栏)。**稳定排序键 = `created` 日期 + `slug` 的 UTF-8 码点序**(不依赖 locale collation;R4-15 后 slug 可含中文,码点序仍确定,M-10「纯 ASCII」措辞随之作废、实质保留),规避 CJK/混排排序跨平台产出不同字节的脚枪(M-10)。
- `check`(源结构一致性,生成式档):①frontmatter 齐备合法;②文档 `line` 引用的 `lines/<slug>.md` 实体存在且未归档;③文档 `id` 全局唯一;④权威/生命周期不变量(见下);⑤worklogs 归档↔登记双向一致。环境自免疫:无 collation、无着色、行尾归一(沿用 Scrollery 已验证纪律)。

#### source universe(P2 阶段 1 定案,2026-07-16;机器真源 = `src/lib/docmeta.mjs` 的 `classifyFile`)

**这张表回答「哪些文件参与哪些检查」。** 它必须先于阶段 3/4 存在:图不变量与生成器都要问「扫描域是什么」,各问各的就会漂移出两个答案。

| 域 | 1a 断链 | 1b 引用可达 | frontmatter 门 | id/取代图(阶段 3) | 索引生成(阶段 4) |
|---|---|---|---|---|---|
| `<docsDir>/` 活区(非 trio、非 archive) | ✅ | — | ✅ | ✅ | ✅ |
| `<docsDir>/{planning,worklogs}/**/{task_plan,findings,progress}.md` | ⬜ 豁免 | — | ⬜ 豁免 | ⬜ | ⬜ |
| `<docsDir>/worklogs/**/closeout.md` | ✅ | — | ✅ | ✅ | ⬜(台账非知识本体) |
| `<docsDir>/archive/**` | ⬜ 豁免 | — | 仅横幅 | **声明了 `id` 才参与** | ⬜ |
| `<docsDir>/lines/**`(阶段 2) | ✅ | — | ✅ | ✅ | ✅ |
| `<docsDir>/status/**`(阶段 4) | ✅ | — | ✅ | ✅ | ✅(聚合进 `STATUS.md`) |
| `sourceRoots` 各目录 | — | ✅ | — | — | — |
| **仓根文件**(R4-11) | — | ✅ `sourceRoots: "auto"` 时 | — | — | — |

- **过程件为何全域豁免**:三件套是**未蒸馏**的工作记忆,收口时被蒸馏为永久文档后即冻结。对它验 frontmatter 等于要求草稿具备成品契约;它也不该进索引——索引的对象是知识,不是草稿。
- **archive「声明了 `id` 才参与」**(阶段 1 定案)。取此规则而非「archive 一律补全 frontmatter」,理由有三:①**零迁移**——Scrollery 实测 28 篇归档件是横幅式、无 frontmatter,强制补全即凭空造一座迁移悬崖,而这些文件的价值恰恰在于「冻结不动」;②**自然正确**——归档是 `git mv`,frontmatter 随文件走,故正常归档的文档**自带 `id`**、自动继续参与唯一性(`id` 必须跨时间唯一,否则一个 supersedes 引用会变得有歧义);③**不变量自愿加强**——补 frontmatter 的归档件自动入图,不补的只需横幅。
  > **已知空洞,诚实记录**:删掉归档件的 `id` 行即可让它退出图。这个洞小且可检出——若有 active 文档 supersedes 它,阶段 3 的悬垂引用检查会红;且按 D-015(只验快照、不读 history),门**无法**知道「它以前有 `id`」。不为此预支复杂度。
- **仓根文件(R4-11 结案)**。原盲区:1a 只走 `<docsDir>/` 之下,1b 的 `sourceRoots` 解析只收**顶层目录**(`readdirSync(root).filter(isDirectory)`)——于是仓根的 `README.md` 里写 `docs/gone.md`,**两道门都看不见**。裁决:`sourceRoots: "auto"` 时**并扫仓根文件**(非递归,`<docsDir>` 自身除外);显式数组时不扫,但可用 `"."` 显式点名仓根(如 `["src", "."]`)。理由:`auto` 的用户意图是「扫我整个仓」,漏掉仓根文件是**实现细节泄漏成语义**;显式数组的用户意图是「只扫我列的」,替他加目录才是冒犯——`"."` 给他一个说得出口的写法,而不必把「文件」硬塞进一个叫 `roots` 的数组。
- **权威与生命周期不变量(R2-C3)**:
  - **状态机**:`draft → active → (snapshot | superseded | archived)`;`superseded` 必须携 `supersededBy`,`supersedes`/`supersededBy` 成对且双向一致(门禁验)。
  - **⚠️ 门只验当前快照,不读 git history(R5-C6;2026-07-16 已裁 → D-015)**。上面这张转换图是**写操作的约束**,不是 CI 的校验对象:仓的当前快照里只有「这个文件现在是什么状态」,**没有「它上个 commit 是什么状态」**——要验「`draft → active` 这一步合法」必须引入 git diff/history 作输入,那是另一台机器(diff-aware validator),P2 不建。**P2 门验的是状态组合的合法性**(状态值∈枚举、终态字段齐备如 `superseded` 必携 `supersededBy`、成对关系双向一致),转换图交由 `upgrade`/未来写命令在**产生**状态变更时约束。理由:快照不变量幂等、可在任意 checkout 上重跑、不受浅克隆/squash merge/rebase 影响;diff-aware 校验则会随 CI 的 fetch depth 与合并策略给出不同答案——把「文档是否自洽」变成「CI 怎么 clone 的」。
  - **权威唯一**:每 `(line, authorityScope)` 至多一个 `status: active` 且 `authoritative: true` 的文档;`authorityScope` 缺省 = 整线,可细分。
  - **改名**:文件名可变、`id` 不变(引用与取代关系挂 `id`,历史引用免疫改名);同 commit 修仓内链接(断链门兜底)。
  - **拆线/并线**:`lines/` 实体文件显式新建/归档(归档线实体留横幅指去向),受影响文档的 `line` 字段同 commit 迁移;归档线不得再被 active 文档引用(门禁验)。

### §7.4 配置化(可配边界:开发期不冻结)

一个 `.<tool>rc`(或 `<tool>.config.mjs`)声明本项目的 taxonomy 与 schema,让门禁不写死:

```jsonc
{
  "schemaVersion": 1,        // 配置/数据布局版本,upgrade 迁移按此逐级走(R2-M3)
  "lang": "zh",
  "dirs": ["designs","reviews","decisions","runbooks","planning","worklogs","archive"],
  "status": ["draft","active","snapshot","superseded","archived"],   // 机器面 ASCII(R3-1)
  // types 仅描述「受治理的入库文档」;INDEX/STATUS 生成物不入库(C-3),不在治理域,
  // 故无 index/rolling-status 型生成物条目——保留 index 型仅用于不变量门档的手工索引文件(R3-8)
  "types": ["design","review","decision","runbook","canon","working-memory","closeout","index"],
  // disposition 是结构化规则,不是裸名字(R2-C5):元模型字段集固定,规则内容开发期可增改(M-6)
  "dispositions": [
    { "name": "experience",   "targetKind": "docs",       "dedupeEvidence": true },
    { "name": "decision",     "targetKind": "docs",       "dedupeEvidence": true },
    { "name": "design",       "targetKind": "docs",       "dedupeEvidence": true },
    { "name": "runbook",      "targetKind": "docs",       "dedupeEvidence": true },
    { "name": "completed",    "targetKind": "docs",       "dedupeEvidence": true },
    { "name": "todo",         "targetKind": "fixed",      "target": "docs/todo.md" },
    { "name": "code",         "targetKind": "frozen-ref" },
    { "name": "test",         "targetKind": "frozen-ref" },
    { "name": "no-promotion", "targetKind": "none",       "reasonRequired": true }
  ],
  "sourceRoots": "auto"      // 或显式列出代码根,供 check-docs 1b 扫描
}
```

`targetKind` 枚举 = `docs`(仓根相对路径,验存)/ `fixed`(固定靶点,配置写死单一路径)/ `line-status`(**按线求解的固定靶点**,见下)/ `frozen-ref`(`@commit` 冻结引用,验 grammar)/ `none`(须 N-A 理由)——validator 按 kind 分派校验逻辑,语义与 §7.1 schema 一一对应。全部 `repo:` 路径共用同一条仓根 containment 断言(§7.1)。

**`line-status` 裁决(R5-C4;2026-07-16 → D-014)**:生成式档把滚动状态从单文件 `docs/todo.md` 分片成 `status/<slug>.md` 后,**`fixed` 表达不了它**——`fixed` 的 target 是配置里一个死字符串,而分片后的靶点随「候选属于哪条线」变化。故 schema v2 新增专用 kind:

```jsonc
{ "name": "todo", "targetKind": "line-status", "statusDir": "docs/status" }
```

配置只声明**根目录**;validator 从该 closeout 所属归档任务三件套的 `line` 字段解析出唯一 slug,求得期望靶点 `repo:<statusDir>/<slug>.md`,再比对 + 验存。**为什么不上通用字符串模板**(如 `target: "docs/status/{line}.md"`):模板引擎会把元模型扩成小型 DSL,可判定性随即外包给字符串插值——validator 将无法在不执行模板的前提下回答「这个 target 合法吗」。专用 kind 把求解逻辑留在 validator 里,元模型只多一个槽位。**为什么不降级成普通 `docs` kind**:那样任何存在的文档都能冒充滚动状态源,固定靶点的强度(「todo 候选必须落在**那一个**文件」)全部丢失,`todo` 与 `experience` 再无区别。

**M-6 裁决:开发期不锁死,保持可配**。自审曾建议把 taxonomy/status/dispositions 锁死以强化「标准纪律」身份(Prettier 式有主见);用户按「**开发期不冻结契约**」原则否决——先放开可配、收集真实使用形态,到**发布**时再据证据决定哪些收敛为不可配的「标准」。**可配的是规则内容(增删改条目),不可配的是元模型**(字段集与 targetKind 语义)——否则「保持可配」退化成 validator 读不懂的任意字符串,可配置性反噬核心不变量(R2-C5)。✅ 配置格式已裁(R2-C5 连带):**JSONC + JSON Schema 校验**,不选可执行 mjs——配置不该图灵完备,静态可校验优先。

**D-003 裁决(2026-07-16):M-6「保持可配」与 P2「契约收敛」的边界 = 元模型收敛,实例保持可配**。二者曾直接冲突——「保持可配」允许随意改 taxonomy,「契约收敛 + `schemaVersion` 1→2」则要求契约定死(编号的意义就是「变了要升版迁移」);**一个字段不可能同时「随便改」和「改了要迁移」**。裁决把 R2-C5 已在 disposition 上用过的解法**平移到 taxonomy**,不发明第二套心智:

| 层 | P2 是否收敛 | 变更后果 | 例 |
|---|---|---|---|
| **元模型**(字段集、类型、语义槽位) | **收敛**,并由 `schemaVersion` 编号 | 变更 = 升版 + 迁移命令 | 「`types` 是字符串数组」「每个 type 带 `canBeAuthoritative` 布尔」「`targetKind ∈ {docs,fixed,line-status,frozen-ref,none}`」 |
| **实例**(具体条目内容) | **不收敛**,保持可配(M-6 存续) | 随便改,**不升版、不迁移** | 你的项目有哪几个 type、叫什么名、哪个能当权威 |

三处原冲突随之消解:①**`types` 枚举**——实例可配,但「哪些 type 能当权威」由元模型的 `canBeAuthoritative` 槽位承载,§7.2「权威唯一」不变量据此判定,不再依赖 type 名字的语义猜测;②**`schemaVersion` 给谁编号**——只给**元模型**编号(schema 第 13 行「配置/数据布局版本」据此解读为**元模型层**的版本);用户增删 `types` 条目**不是**布局变更、不触发升版;③**Scrollery taxonomy 与本仓不同**(实测:`docs/` + 54 个 `line` 语义值 + 字母登记表 vs 本仓 `docs/` + designs/planning/worklogs)——正因实例可配,二者可共用同一元模型而各配各的实例,无需削足适履。

### §7.5 采纳梯度(C-1:降低团队采纳阻力)

纪律型工具的头号死因是被贡献者当官僚主义拒收(`--no-verify` 绕过 / fork 掉门禁)。为让它能**渐进采纳**:

- **两级门禁**:①**核心不变量**(断链、归档横幅、收口契约、schema 合法);②**风格建议**(命名、摘要长度、可选字段)。
- **三档开关 + 显式 profile(R2-M1)**:每类门可配 `off | warn | enforce`,但默认值**只经显式 profile 取**——规则级别与「新仓/存量仓」不再各自隐式覆盖(消解 v0.2「核心默认 enforce」与「新库从 warn 起步」的自相矛盾;`init` 选 profile,测试按 profile 断言退出码):

| profile | 核心不变量 | 风格建议 | 适用 |
|---|---|---|---|
| `strict` | enforce | enforce | 纪律成熟团队;本仓 dogfood;**新仓亦用此档** |
| `brownfield`(存量仓默认) | 存量违规入 **baseline 豁免清单**,仅**新增**违规 enforce | warn | 存量仓渐进采纳,免「一上来满屏红」劝退 |
| ~~`greenfield`~~ | — | — | **已砍(D-002,2026-07-16)**:新仓无存量 → baseline 为空 → 行为与 `strict` **完全相同**,是同一档的两个名字 |
| ~~`advisory`~~ | — | — | **已砍(D-002)**:全 warn 是**输出级别**不是**采纳档位**,与 baseline 概念正交 → 降为全局标志 `--warn-only`,不占 profile 语义位 |

**D-002 裁决(2026-07-16):四档砍为两档 `strict` / `brownfield`,`advisory` 降级为 `--warn-only` 标志**。三条理由:①**实测代价**——`profile` 当前是**空壳**(`grep profile src/` 全仓仅 1 次命中,即 `config.mjs` 的 DEFAULTS 赋值,**无任何代码读它**);对 Scrollery `dev` 实跑 `worklog check` 并声明 `"profile": "brownfield"`,仍 **exit=1**、全量报红——**配置说了话,代码没听**,而这正是 §7.5 开篇「被当官僚主义拒收」的具体形状。②**概念冗余**——`greenfield` 与 `strict` 在新仓行为同一;`advisory` 是输出级别而非档位。③**主路径已定**(L11/B 裁决)——Scrollery 是首要消费者且是存量仓,`brownfield` **不是可选档,是主路径**;worklog-kit 自身用 `strict`。两档覆盖 100% 已知需求,工作量减半。
> **排期含义**:profile 须**前移**至 P2 早期(原排阶段 6)。阶段 2 的线引用门一上,存量仓会在既有红之上再叠一层线引用红;**baseline 必须先在**,否则每个后续阶段都在给一个已经满屏红的仓继续加红。

**baseline 承载(R4-16;2026-07-16 已裁)**:
- **载体**:独立仓根文件 `.worklog-baseline.json`,**入库、受治理**(它改变门禁判定,必须可 review、可 blame,与配置同级)。
- **条目钥匙** = `<仓根相对路径> + <规则 key>`(如 `docs/old.md + docs.statusInvalid`),可选内容 hash 增强;**文件改名即失配 → 按新增违规 enforce**——取保守方向:改名是主动触碰,顺手清账合理,豁免不跟随移动。
- **再生成**:`worklog baseline --update` 显式重扫落盘;**check 永不自动吸收新违规**(否则 baseline 退化为「自动豁免一切」)。
- **判定语义**:`brownfield` 档下 check 对每条违规先查 baseline——命中即豁免(计数单列报告,不静默),未命中 enforce;`strict` 档无视 baseline。
- **适用域(R5-C5;2026-07-16 已裁 → D-013)**:**baseline 只豁免可稳定定位的 per-file 存量规则**(frontmatter 非法/缺横幅/断链等,主语是「某个文件」);**P2 的图不变量永远 enforce,不可豁免**——双权威、`id` 重复、`supersedes` 断边、归档线被 active 文档引用,这些违规的**主语是文档集合/ID/line,不是某一个文件路径**。「路径 + 规则 key」这把钥匙接不住它们:随便挑一个参与文件当钥匙,则文件顺序或改名会改变 baseline;给每个参与文件各记一条,则删掉一端后残留**假豁免**;用消息文本 hash,则改一次文案或加一门 i18n 就全量失配。若将来确有豁免图违规的业务需要,**须为每条图规则单独定义 canonical subject**(如 `line:<slug>|scope:<scope>`、`doc-id:<id>`),不得复用万能钥匙。
  > **落地推论**:`brownfield` 装进存量仓的第一天,双权威这类红**照样红**。这是有意的——图不变量正是 P2 的产品价值所在,「存量太多先记账」对它不适用:双权威不是历史包袱,是**现在就在误导读者的活缺陷**。

**采纳步骤(brownfield;2026-07-16 靶场实测后补,顺序不可换)**:

1. **先按你的 taxonomy 配置**(`dirs` / `status` / `types` / `dispositions`),**再**跑门。跳过这步直接开跑,满屏红里绝大多数是「工具不认识你的枚举」而非「你欠了债」——**实测三轮:118 → 15 → 3 条,差额全部来自配置,靶场一个字节没改**(两次同型错误:先猜 type 枚举得 83 条假红,后少声明 disposition 得 12 条假红)。这正是本节开篇「被当官僚主义拒收」的最短路径:第一次跑就给人一屏与他无关的红。
2. `worklog init --profile brownfield`(或让它自动判:stamp 前 `docsDir` 已有 `.md` ⇒ brownfield)。
3. **`worklog upgrade`——爬第一把梯子**(P2 阶段 1 补,2026-07-16)。它会把机器**派生得出**的字段自动补齐(当前即 `id`:按 `<created>-<文件名 slug>`)。**这步不能跳、也不能用 `baseline` 代替**:`id` 缺失按 D-013 结构上**不可入账**——它不是「工具修不了的旧债」,一个命令就能真修好,给它开豁免等于用记账挂起替掉一条走得通的路。
   > **⚠️ 反直觉之处(实测挖出,勿按「版本号已最新就没事」理解)**:`init` 给存量仓 stamp 的配置**就是最新版**,故这类仓**一次迁移都不需要,却极其需要这一步**。`upgrade` 管的是「让**数据布局**与所声明的 `schemaVersion` 相符」,不是「改那个数字」——版本号对了不代表数据到位。它对已到位的仓幂等、零写入。
4. `worklog baseline --update` 显式立账,**review 这份 diff 再 commit**。此处清的是**另一把梯子**才够得着的那类:值属人工判断、机器派生不出(如 `line`——机器不知道一篇文档属于哪条工作线)。
5. `worklog check` 应绿;此后新增违规照红,旧账留着但不许在同一处再欠新的(棘轮)。

> **两把梯子,别搞混**(§4.1 item1 已立为判据):第 3 步与第 4 步**不可互换、也不可省其一**。判据只有一条——**机器能否派生该字段的值**:能则 `upgrade` 自动补(`id`),不能则 `baseline` 立账挂起(`line`)。派生依赖 `created`,故若有文档连 `created` 都没有,`upgrade` 会**整体拒绝、零写入**并报出名单(两条出路:补 `created`,或直接手填 `id`)——不做「能播的播、播不了的留着」,因为半播种的仓里没有人知道该给剩下的发什么号。实测靶场 `created` 零缺失,此路对它畅通。

> **实测结论(靶场 = Scrollery `dev` `d7200f1` 只读副本,165 md / 8 类型目录 / 3 个月演进)**:配置对齐后真实存量债 = **3 条断链**;立账后 `check`/`index` 双 0,新增违规 → 1,`--warn-only` → 0。**⚠️ 此数不可读作产品成熟度**:Scrollery 是这些不变量的**抽取源**,是最好情况而非代表性存量仓——它的 taxonomy 就是本工具 taxonomy 的原型。「3 条债」证明的是**抽取忠实**,不是「能处理随机存量仓」;后者仍无证据。

- **P1 单 profile(R3-11)**:MVP 只内置 `strict`(dogfood 自用即此档),`strict`/`brownfield` 两档与 `--warn-only` **已于 2026-07-16 阶段 0 落地**(此前两轮实测均为空壳:`grep profile src/` 只有 DEFAULTS 赋值一处,声明 `brownfield` 仍全量报红)——梯度**语义**在 v0.3 定死,**实现**未抢 MVP。
- 「能被**渐进**采纳」提为**显式设计目标**,与「机械强制」并列——强制是终态,不是入口。

## §9 CI 设计

GitHub Actions;`init` 产出的 `ci-github.yml` **按 profile 生成**(R2-M4),不是一份通吃:

- **门禁步**(逐条 bare 跑,分别检查退出码——不用 shell `;` 链吞码):`check-docs`;生成式档加 `index check`(生成成功 + 源结构一致性)。**调用须钉精确包版本**(`npx --yes worklog-kit@<ver>`,D-017),不用裸 `npx worklog`——版本浮动 = 门禁判定浮动。**不含 `install-skills --check`**——个人 skill 安装一致性属本机 `doctor` 检查,CI 的 fixture HOME 里没有个人副本(R2-M4;修正 v0.2 唯一事实不符项——Scrollery 现役 CI 本就只跑 `--selftest`,v0.2 的「同构」声称与被抽取系统自身设计意图相悖)。governance-only profile 不含任何 skill 步。
- **selftest 步**:单条 `worklog selftest`(聚合 config + 各门 fixture + e2e)。**聚合器是刻意的**:新增套件只须登记进 `SUITES`,不必再同步改 CI 模板与 npm 脚本——「加了套件但 CI 没跑」是那种只在事后才发现的漏。各门仍保留自己的 `--selftest` 供定点重跑。
- **发布步(生成式档,P2+;R3-2)**:`index build` 产物上传 **Actions artifact**(最低承诺);docs site 发布 opt-in。
  - **D-006 裁决(2026-07-16):P2 阶段 7 判据拆两半,先验本地能验的**。原判据「artifact 发布可复现」须 CI 产出,而本仓 CI **billing 阻断**(run `29209040905`,4s failure)→ 按 L12 元判据属**不可测**。拆解:①**「生成器幂等 + 产物可复现」= 本地可验**,走 **WSL 自托管 runner**(用户环境就绪,Scrollery 侧已跑通多次,见其 `runbooks/2026-07-06-CI自托管runner运维手册.md`)→ **计入 P2 判据**;②**「GitHub Actions 集成可用」= 本地验不了**(本地 runner 绿 ≠ hosted 绿:runner 镜像/权限/`merge_group` 事件注册均不同)→ **移出 P2**,待 billing 恢复(预计下月)后补验。**本地绿不等于 hosted 绿,此处不得混为一谈**。
- **e2e 步**:在临时目录 `init` 一个假仓 → 跑全部门应绿(见 §10)。
- **主干兜底与要件(R2-C6)**:`main` push 触发同一套门 = **事后探测 + 恢复兜底**,不是拦截;多人档要件 = required status checks + up-to-date-before-merge 或 merge queue——CI 模板对 `merge_group` 事件注册同一套门,在**拟合并结果**上跑。
- **跨平台**:至少 ubuntu + windows(Scrollery 已验证 Windows 行尾/路径坑,门禁须行尾归一)。
- **Node 矩阵**:LTS 双版本。
