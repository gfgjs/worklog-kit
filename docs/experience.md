# 经验账(收口蒸馏落点;条目按 F-ID 锚定,追加式)

## 直证 probe 对着设计抄,不对着实现抄(F-015,2026-07-16)

「设计表 ≡ 代码表」的直证 selftest 只证得了「代码 ≡ probe」——probe 若照实现落笔,三处会**一致地错**(source universe 的 closeout `index` 格:设计 ⬜、代码 true、probe 照抄 true,潜伏到生成器第一次真读这一格才暴露)。两条守则:①写直证时对着**设计正文**抄;②没人读的格连直证都护不住——**新增消费者上线前,重核它要读的每一格**。同型病:被关掉的检查会掩护它自己的 bug(F-008)。

## 坏盘取证三则(F-017,2026-07-16;D 盘中途损坏、系统重启、仓库迁 C)

1. **所有权门实战首咬**:迁移后 `.worklog/generated/manifest.json` 是 244 字节全 NUL(坏盘写残——长度对、内容清零),R5-M7 门以「缺 marker 拒绝覆盖」硬拦;任何「存在即跳过/直接覆盖」策略都会静默吞掉这类文件。
2. **生成物不入库 ⇒ git 完整性检查天然不覆盖它**;「删掉重建」就是这类文件唯一正确的完整性策略——重建后 hash 与损坏前记录逐字节一致,顺带复证跨盘决定论。
3. **push 客户端报错 ≠ 远端没动**:坏盘期两次 push 报 `sideband disconnect`,实则服务端已生效(断在读响应、不在传包)。push「失败」后先 `git ls-remote` 复核远端真实状态,再决定重试——否则会把已上行的提交再推一遍并误报事故范围。另:盘坏前的 EUNKNOWN/ENOSYS/幽灵 `garbage found` 告警全是前兆,勿一律归咎杀软。

## WSL 自托管 runner 运维(F-016,2026-07-16)

- runner 是**仓级**作用域:一台机器服务两个仓 = 两个独立目录、两次注册(本仓 `~/actions-runner-worklog`,与 Scrollery 实例并存)。
- `nohup ./run.sh` 裸跑**死于宿主机重启**(当日实证)。服务化 `./svc.sh install <user> && ./svc.sh start` 需 root;**无 sudo 密码时用 `wsl -u root -e bash -c '…'`**(WSL 允许 Windows 侧直接以 root 启动发行版,免密)。服务名:`actions.runner.gfgjs-worklog-kit.dev-box-wsl-worklog.service`。
- 杀旧实例的 `pkill -f` 模式会匹配到**自己所在的命令行**把自己毙掉——用 `[r]un.sh` 字符类写法自免疫。
- 非交互 shell 不加载 `.bashrc`(顶部早退),nvm 装的 node 须补进 `~/.profile` 才能被 `bash -lc`/CI 步骤看见。

## 修一个报障点要扫同型全部落点(F-020,2026-07-17)

一个缺陷形态常有多个落点。D-017 裁「CI 模板不教裸 `npx worklog`」当时只改了 CI 模板并加 e2e 哨兵防回归,README 快速上手的同句教程没跟改——npm 上真有他人同名包 `worklog@0.1.9`(实测:消费仓裸跑提示装它),新用户照抄 README 第一条命令即抓陌生包执行。同轮还带出「共 11 套」套数漂移(doctor 第 12 套后加、散文没跟)。两条守则:①裁决/修复落地时**全文检索同一错误形态的全部落点**(模板 / README / runbook / 散文),不是只修报障那一处,否则留同病兄弟潜伏到下个用户;②机械哨兵有覆盖面——e2e 哨兵只咬结构面(模板 fixture),咬不到散文教程,散文正确性仍靠全文检索兜。同型病:被关掉的检查会掩护自己的 bug(F-008)、直证 probe 对着实现抄会三处一致地错(F-015)。

## 三件套体积口径须双轨:行数对中文文件失真(F-025,2026-07-19)

案例实证(scrollery 备份施工线,任务过半):task_plan/findings/progress 三件 110/48/119 行,全低于 F-021 定的行数护栏(200),但字节 23.7KB、est ~6.8k token/次全量读——用户主观已判税重。行数护栏对中文密集文件天生失真(中文一行的计费重量远高于同长度 ASCII)。治理:doctor `fatTrio` 改双轨——行数 200 兜底(ASCII 长文件仍拦)+ 单件 est>4k + 三件合计 est>6k(estTokens 与 `tools/token-audit.mjs` 同式:CJK=1/字、其余=1/4 字符)。全 warn 不 fail,同 F-021/F-022 立场不变。

## 三件套冗余三源:治理面在写侧不在读侧(F-026,2026-07-19)

案例解剖坐实同一信息重复落 3-4 处:①task_plan 已 complete 阶段仍留全部 checklist 实现细节(commit 已记,占该文件过半);②progress 每会话段「做了/遗留」复述 task_plan 阶段定义与 commit message;③findings 侦察/调研节落地后仍全文常驻。三处的共同病根是**写时不折叠、指望读时省**——F-021 已否 A1(接续跳读 findings 是负优化,踩坑一次返工远超省的读)证明读侧裁剪路线不可行。治理方向必须在写侧:阶段完成当场折叠 checklist、progress 写指针不写复述、findings 消费当场标记。写时标记与 A1 的读时跳读风险不同——前者是落地当下的确定知识,后者是接续时的猜测,不能混为一谈裁决。

## 模板结构须自证 append-only:收口专用节不得挡文件尾(F-027,2026-07-19)

省 token 写侧纪律引入「纯追加走 shell heredoc(`cat >> 文件`)」,本线自身 dogfooding 时当场撞坑:`progress.md` 模板把「回顾(收口时填)」节放在文件最末,盲追加的新会话段被粘到回顾节之后,段序错乱。修法:回顾节移到「前情」段之后、会话段之前——文件尾永远留给最新一次追加。同型病:任何「仅特定时机才写」的节都不能占文件尾位置,否则与常态追加互斥,是模板设计的通用检查项。

## 省 token 砍文件体积而非读次数(F-021,2026-07-17)

token 税实测(读 60% / 写 35% / skill 注入 1% / CLI 输出 0.03%)后定位:读税大头是 harness「Edit 前置 Read」强制的重复读,**不是**接续会话的重复读——31 个 kit 会话均 30+ 读/会话,远超接续频次。故杠杆在**文件体积**不在读次数:计费乘数(prompt fold → billed ×2–5)让每减一行都乘倍进账。三条推论:①优化面在提示词/模板层,引擎层税已近零;②接续跳读 findings 是**负优化**(踩已记录的坑一次返工 10k+ 远超省的 1.4k,故否 A1);③三件套行数软护栏是治本梯子——doctor `TRIO_WARN_LINES=200`(warn 不 fail,避免逼施工中途压缩丢证据),实测正常任务 ≤134 行、失控样本 211–309。
