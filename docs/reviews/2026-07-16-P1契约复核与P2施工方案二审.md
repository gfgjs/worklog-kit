## §2 阻塞项(🔴)

### R5-C1 · closeout 存在 false negative 与仓根边界逃逸

**证据**:

- `src/check-docs.mjs:85-89` 在 `targetKind === 'none'` 分支末尾直接 `continue`,全局 `verified === 'yes'` 校验位于 `:103`,故 no-promotion 永远绕过 verified。
- `src/check-docs.mjs:93-100` 仅用正则检查 `repo:<路径>` 后直接 `join(root, path)` 验存;未拒绝 `..`、绝对路径,也未用 `resolve` 后断言仍在仓根内。
- `src/check-docs.mjs:73-78` 只认处置表首列含 candidate ID,随后按列位置解构;其余六个列名、列数、顺序均未校验。

**影响**:核心卖点「提升动作已同变更集完成、落点在本仓、固定 schema 可机械校验」并不成立;恶意不是前提,一次误填路径即可把仓外文件当合法 target。

**建议**:先做统一 `validateRepoRef()`(拒绝绝对路径/空段/`.`/`..`,`resolve` 后做 containment,必要时规定 symlink 语义);把 verified 校验移到 disposition 分派之前或 finally 式公共尾;严格校验表头/列数。三类各补正负 selftest。

### R5-C2 · cardinality 规范分叉,实现者无唯一答案

**证据**:

- 设计 §7.1 写明「一候选提升到多处 = 同 ID 多行,以 `(ID, disposition, target)` 唯一」。
- `src/check-docs.mjs:76-81` 的 `seen` 仅按 ID 去重,第二行必报 `closeout.reDisposed`。
- `skills/planning/SKILL.md:60`、`templates/closeout-template.md:10`、根 README `:39` 均写「每候选恰好一行」。

**影响**:这是 contract split,不是单改代码即可收口;按设计修会破当前 skill/template/README,按实现修需改设计唯一真源。

**建议裁决(倾向)**:**一候选恰好一处置行**。若一个发现确实要产出 design + test,拆为两个原子候选并在摘要互链。此形态与现行实现/用户工作流一致,覆盖集合和去重语义最简单。若保留一对多,则 `seen` 必须改组合键,同时另按 ID 集合做 coverage,所有模板/skill/README 同步改写。

### R5-C3 · schemaVersion 先升版、upgrade 后补,重演“只上门不给梯子”

**证据**:

- P2 task plan 阶段 1 交付 `schemaVersion 1 → 2` + registry **骨架**;阶段 5 才交 `upgrade --dry-run` 与升档 e2e。
- 设计 §12 P2 的完成判据已经要求升档迁移 e2e,§12 P4 却又把 `upgrade`/migration registry 当 P4 交付。
- 设计 §6 已裁 break schema 必须附 migration、原子写、备份回滚。

**影响**:阶段 1 一旦合并,存量仓会先看见 schema 2 门,却拿不到支持的迁移入口;后续四阶段都在未完成升级契约上叠功能。与 F-001“引用门与迁移命令同批”完全同病。

**建议**:把 **schema v2 schema + 运行期 validator + migration registry + `upgrade --dry-run` + v1→v2 fixture + 原子/备份基础设施**合为一个原子阶段。P4 只做已存在 upgrade 能力的公开分发完善,不再首次引入 upgrade。

### R5-C4 · status 分片无法由现有 disposition 元模型表达

**证据**:

- 当前 todo 规则是 `targetKind: fixed,target: docs/todo.md`(`src/lib/config.mjs:23`)。
- P2 task plan 阶段 4/5 把滚动状态迁成 `status/<line>.md`,并要求 skill 按档改靶点。
- 设计 §7.4 的 `targetKind` 仅 `docs|fixed|frozen-ref|none`;“fixed 档位化”没有可执行 schema。一个固定字符串不能随候选所属 line 变化。

**影响**:生成式档 closeout 无法机械验证 todo 候选究竟应落哪个 status 文件;若退化为普通 `docs` target,任何存在的文档都能冒充滚动状态源,丢失固定靶点强度。

**建议裁决(倾向)**:schema v2 新增专用 `targetKind: line-status`,配置只声明根 `docs/status`,validator 从归档三件套的 `line` 解析唯一目标并验存。不要上通用字符串模板引擎,避免把元模型扩成小型 DSL。

### R5-C5 · baseline 钥匙只覆盖单文件违规,接不住 P2 图不变量

**证据**:D-008 将钥匙裁为「相对路径 + rule key」;P2 随后新增重复 ID、双权威、`supersedes` 断边、归档线被 active 文档引用等跨文件违规。这些违规的主语是文档集合/ID/line,不是唯一文件路径。

**影响**:若随意选一个路径作为钥匙,文件顺序/改名会改变 baseline;若给每个参与文件各记一条,删一端后可能残留假豁免;若用消息文本 hash,文案/i18n 改动会导致全量失配。

**建议裁决(倾向)**:**baseline 只豁免可稳定定位的存量 per-file P1 规则;P2 图不变量永远 enforce。**若业务确实要豁免图违规,须为每条规则定义独立 canonical subject(如 `line:<slug>|scope:<scope>`、`doc-id:<id>`),不能复用“路径 + rule”万能钥匙。

### R5-C6 · 实体 schema、扫描宇宙与“状态机校验”尚未可实现

**证据**:

- `lines/<slug>.md` 只规定“一句话使命 + 可选 owner”,未定义其 frontmatter/type/id/status/归档形态。
- `status/<line>.md` 仍写“每工作线一文件(或每任务一文件)”,task plan 却直接按每线施工;两份规范分叉。
- 设计 §7.2 生成扫描列表只列 designs/reviews/decisions/runbooks/worklogs;但全局 ID、`supersededBy`、归档线引用检查必须能看见 archive/lines/status。
- “`draft → active → ...` 状态机”对当前仓快照只能校验**状态组合**,无法知道某文件上个 commit 是 draft 还是 active;要校验转换必须引入 git diff/history 输入,现方案未设计。
- D-003 要求 type 实例带 `canBeAuthoritative`,当前 schema/task plan 阶段 1 仍是字符串数组,且字段清单漏 `authorityScope`。

**建议**:开码前冻结三张表:①实体类型及必填字段;②source universe(哪些目录参与 ID/引用/生成,哪些只显示不索引);③snapshot invariant。建议 CI 只验**当前快照**(唯一性、引用、成对关系、合法状态组合),状态转换图用于 `upgrade`/未来 task CLI 的写操作约束,不在 P2 引入 diff-aware validator。
