# worklog-kit Token 优化后产品与代码深度复审

> **复审对象**：私仓 `409a404`（上一份综合报告入库）至 `d7a0302`，公开仓单基线 `055c405`，npm registry，及 Scrollery `dev` 工作树。
>
> **核心问题**：本轮 token 优化是否真实、省在哪里、是否以施工质量或安全为代价；worklog-kit 对 Scrollery 是否仍值得；产品和代码下一步应收缩还是扩张。
>
> **公开边界**：本文是 private review，含内部发布状态、消费仓采用数据和安全审计结论；若入库，须与 public export 排除规则同批处理，不得依赖“以后记得排除”。

## 0. 结论先行

### 0.1 总裁决

**继续使用、继续做，但立即收窄。**

worklog-kit 对 Scrollery 的核心价值仍成立：跨 session working memory、逐候选 closeout、以及能真实拦错的 docs gate。真正值得长期投入的不是“再造完整 SDD 平台”，而是：

> provider-neutral、repo-native 的 engineering-memory compiler：把施工中的易失状态压成可恢复 checkpoint，再把耐久发现逐候选提升进永久知识。

但本轮不能宣称“已经把 token 税砍掉 50%”或“Scrollery 已经变省”：

1. 新 skill 和模板**尚未部署到 Scrollery**；现有 24 个 active task 里，`前情`采用数为 0，`progress-archive.md` 为 0。
2. 新 skill 自身从 2,239 增至 2,735 token，三模板骨架从 570 增至 768；新任务 upfront 税先增加 694 token。
3. 对 Scrollery 存量做反事实分层读，active trio 合计可降 22.2%，但中位任务只省 146 token；64.7% 收益集中在 UIUX 单一离群任务。
4. token audit 无时间/版本分桶，并漏记新版推荐的 Claude heredoc 写入；它会让优化后数字被动变好，却不能证明真实成本同比下降。
5. 新版固定 Bash heredoc 对外部内容存在 delimiter 提前闭合风险，且在 Windows/PowerShell、当前 Codex 编辑纪律下不可移植。

因此本轮最准确的定性是：

> **方向正确的长尾减税方案；尚未部署、尚不可归因、典型任务可能小幅增税。**

### 0.2 当前优先级

| 级别 | 裁决 |
|---|---|
| P0 | 停止用 `0.1.0-alpha.2` 代表当前源码；registry、公仓 tag、私仓 HEAD 已是三套不同内容。下一版前必须恢复“tag → exact packed tarball → registry”可追溯链。 |
| P0 | 暂停任何“已省多少”对外结论；先修 token audit 的时间/版本分桶和 shell write 漏记，再做 A/B。 |
| P1 | 撤回固定 heredoc 作为通用写入主路；改成 `worklog note/checkpoint --stdin` 或 agent-native 安全编辑。 |
| P1 | 把 closeout 815-token 冷章节移出 hot skill，移除 Scrollery 长任务对 `docs/worklogs/README.md` 的默认热读。 |
| P1 | 实现 `start/list/resume/note/checkpoint`，让省 token 成为产品能力，不再依赖提示词技巧。 |
| P1 | 修 Scrollery CI 的 `main`/`dev` 触发错位、2×2 消费矩阵和旧新双门。 |
| P2 | team、generated status、paranoid CI 改为显式 opt-in；没有真实 team 使用证据前不继续扩。 |

### 0.3 是否值得交 token 税

**值得交核心税，不值得交当前全税。**

- 值得：长任务 checkpoint/trio、candidate closeout、frontmatter/link/closeout gate。
- 按需：lines、status、generated index。
- 默认关闭：team event 模型、消费仓全量 engine selftest、跨 OS/Node 矩阵。
- 短任务：不应因“恰好 ≥5 步”自动升级为三件套；更合理判据是预计跨 session/compaction，或高不确定性、高返工风险、多阶段依赖明显。

这 694 token 不是坏事本身；若后续每轮少读几千 token，很快能摊平。但它意味着：

- 一次 session 内结束的边界任务纯亏；
- “空模板变短”不是本轮事实；
- 必须用真实任务分布算 break-even，不能只展示极端长任务。

当前 skill 的 closeout 章节单独约 815 token，占整个 skill 29.8%，而只有用户明确收口时才需要。`templates/runbook-closeout.md` 已有更完整手册，继续把 815 token 常驻 hot skill 是最明显的重复税。

|---|---:|---:|---:|

- 当前 cold resume 中位：`6,243 + 5,217.5 ≈ 11,461`；

### 3.5 历史 token audit 能说明什么

当前 Scrollery 混合历史账约 2,592,281 fold-est token：

| 机制 | 占比 |
|---|---:|
| Edit 前置 Read | 41.2% |
| Write/Edit 输入 | 35.5% |
| 主动/接续读 | 18.3% |
| skill body | 2.7% |
| description | 2.2% |
| CLI | 0.04% |

它可靠支持一个方向性结论：**三件套体积 × 读写次数是主要税基。**

它不支持“本轮节省了多少”，原因见 `tools/token-audit.mjs:49-288`：

1. 无 `--since/--until`、revision、task、skill hash 或 worklog-kit version 分桶；
2. description 依据“现在是否装 skill”追溯计入旧 session；
3. 旧/新用 JSONL 文件 mtime，而不是事件时间；
4. Claude Bash/PowerShell 命中 planning path 一律登记为 read，`>>`/heredoc/`tee` 写入不计命令输入；
5. 现有 Scrollery 日志已抽到 15 次 shell planning write、约 9,854 est-token 输入被漏，其中 heredoc/`>>` 约 8,768；
6. Codex 分支不看 function name，只看序列化 arguments；一条多文件命令只归到第一个命中类别；
7. CLI regex 漏 `node .../worklog.mjs doctor` 等真实调用；
8. 无 audit 自测 fixture；“计费约 ×2–5”也没有账单/cache 数据支撑。

新 skill 正在把 Write/Edit 转去 shell；audit 恰好漏这条路。若不先修，优化后报表会发生系统性指标美化。

### 3.6 Doctor：算法准，输出面反增税

正项：

- `src/doctor.mjs:28-36` 的 CJK/other 估算在 Scrollery 72 个 trio 文件上总量约低估 6%，中位文件误差约 -9.5%；
- 4k/6k 命中集合与 o200k 实测一致；
- 双轨比单纯 200 行明显更适合中文密集文档；
- warn 不 fail 避免为过门而删证据，方向正确。

问题：

- Scrollery doctor 输出 64 行、约 11,087 o200k token；其中 token 类 warning 56 条，诊断本身已超过典型 cold resume；
- 默认逐件输出，没有 top-N summary、`--verbose`、`--json` 或 ratchet；
- `foldableStages()` 只看 H3、complete/done、`[x] > 3`，115 个完成节只报 33，另有 H2 完成节完全不看；
- `doctor.fatTrioDir` 文案仍说“接续每轮全量读这套”，与新版分层读取自相矛盾；
- team `progress/events/` 被设计为“天然不咬”，最可能增长的 event history 没 token 护栏；
- repo==package 的本仓继续固定报 6 个 managed copy missing，噪声仍在。

建议默认只出：总数、top 5、可执行建议、`--verbose` 入口。CI 若要使用，另做 baseline/ratchet；人用 doctor 保持 exit 0 可以。

## 4. “绝不影响质量”的审查

### 4.1 可保留的优化

以下方向能在不删事实的前提下降税：

- `progress` 只热读 `前情 + 最近两会话`；旧会话冷移；
- 已消费侦察材料离开 hot view，但原文保存在 cold ledger；
- complete implementation steps 折叠为 commit/hash 指针；
- 大附件单独存放、按需读取；
- doctor 用真实 token 口径提醒长尾；
- closeout 逐候选处置继续保持强门，不能为了省 token 弱化。

### 4.2 当前不是严格 lossless

`skills/planning/SKILL.md:40-44` 目前允许直接把 checklist 和 consumed findings 压成一行，只有旧 progress 会话明确迁 archive。风险：

- checklist 可能是 acceptance criteria、不变量或用户裁决，不是实现步骤；
- 折叠发生时 commit 可能尚不存在；
- shallow clone、squash、fresh-export 或单文件交接时，Git 指针可能不可达；
- “细节已在 git/progress”没有机械验证；
- `前情` 每会话要改文件顶部，在 Edit-before-Read harness 中仍会触发整文件 read。

折叠规则应明确区分：

- **可折叠**：实现步骤、重复状态、已消费探索过程、长工具输出；
- **不可折叠**：acceptance criteria、不变量、用户裁决、失败边界、未解决错误、最终验证证据。

最稳模型不是改写 canonical ledger，而是：

`append-only lossless ledger → deterministic compact hot view`

即 `worklog checkpoint/resume` 生成或输出有界视图，原账不丢。

### 4.3 固定 heredoc 是安全与结构回归

`skills/planning/SKILL.md:48` 推荐：

`cat >> 文件 <<'EOF'`

存在四类问题：

1. 外部/不可信内容允许进入 findings；若包含独占行 `EOF`，heredoc 会提前结束，后续文本被 shell 当命令解析。
2. 用户任务名只剔除 Windows 非法字符，仍可能含 shell metachar；示例没有形成完整 path quoting 契约。
3. Windows/PowerShell 没有同形 heredoc；当前 Codex 还明确要求文件编辑走 `apply_patch`，不能依赖 shell write trick。
4. `templates/findings.md:19-22` 的 EOF 是候选表；盲追加“发现新行”会落进“耐久提升候选”节，而不是 `## 发现`。

固定 delimiter 即使单引号也挡不住 terminator collision。正确修法不是再加一句“内容当数据”，而是产品化：

- `worklog note <task> --kind finding --stdin`
- `worklog checkpoint <task> --stdin`
- 工具负责 section 定位、EOL、锁/原子写、frontmatter、delimiter 不相关；
- 或 agent-native structured edit；不把不可信文本重新解释为 shell program。

在这项修复前，不应继续推广 heredoc 主路。

## 5. 新增代码与发布发现

### P0-01：同一 alpha.2 三套内容，可追溯链已破

| 表面版本 | pack shasum | 状态 |
|---|---|---|
| npm registry `0.1.0-alpha.2` | `b691023…` | 已发布事实 |
| 公仓 `v0.1.0-alpha.2` / `055c405` | `66d5ed5…` | 与 registry 18 个 package 文件不同 |
| 私仓 HEAD `d7a0302` | `e76c9b3…` | 与 registry 24 个、与公仓 7 个 package 文件不同 |

`package.json:3` 三处仍都是 `0.1.0-alpha.2`。公仓 annotated tag 指向 fresh-export 基线，不再是 registry tarball 对应源码。上一轮曾验证的 surface/tag 一致性，在公史重建后被破坏。

影响：

- 用户无法从 `v0.1.0-alpha.2` 重建 registry 中的 alpha.2；
- source review、SBOM、漏洞定位、provenance 都失去准确锚点；
- 本地 `npm pack --dry-run` 与 registry shasum 不同不再只是“开发中未 bump”，公仓 release tag 也已错误；
- 下一次若沿现流程继续 sync/tag，问题会固化。

建议：

1. **不要再次移动、覆盖或重写已经公开的 tag/history**；处置需用户明确授权。
2. 先定位能重建 registry alpha.2 的私仓 commit/tarball，写清 release discrepancy；不要假装当前 tag 等价。
3. 下一版使用新版本号（如下一 alpha），顺序改为：净化公开树 → 从公开 exact commit `npm pack` → blocklist/allowlist 扫 exact tgz → 发布这个 tgz → 校 registry shasum → tag exact commit → 验 tag pack shasum。
4. 发布 provenance；npm 官方支持生成 provenance statements：[npm provenance](https://docs.npmjs.com/generating-provenance-statements/)。
5. release selftest 应从最终 tgz 安装并 spawn `bin/worklog.mjs`，不再只 import 私仓源码。

### P1-02：`token-audit` 是未经测试的度量核心

`tools/token-audit.mjs` 325 行级、无导出纯函数、无 fixture、自测聚合器未接入。它当前既不能归因，又漏新版写法，却已被用作优化决策依据。

应拆为：

- parser adapters：Claude/Codex 各自 fixture；
- canonical event model：read/write/skill/CLI/session/time/version/task；
- estimator：纯函数，与 doctor 共用或生成同源；
- reporter：`--since/--until/--task/--json`；
- golden tests：shell append、apply_patch、多文件、cache、重复盘符、active session。

### P2-03：发布包携带一个坏 script

`package.json:35` 发布了：

`"token-audit": "node tools/token-audit.mjs"`

但 `package.json:22-28` 的 package `files` 不含 `tools`，`npm pack --dry-run` 的 48 个 entries 也没有该文件。结果是发布 manifest 暴露一个在 tarball 内必定找不到入口的 script。

若它是 repo-only dev tool，应避免把坏 script 带给消费者；若它是产品能力，应纳入 CLI/package 和测试，不要卡在两种身份之间。

### P1-04：team 工作流与新版 resume/doctor 断开

team 转换后删除 `progress.md`，只留 `progress/events/`。但：

- `skills/planning/SKILL.md:55` 接续路径只读 `progress.md`，全篇没有 team/events 分支；
- `src/doctor.mjs:50-72` 的体积护栏只扫三件平文件，明确让 events “天然不咬”；
- 当前 kit/Scrollery 均无真实 team task 使用证据。

这是功能边界未闭环，不宜继续默认安装。要么：

- 让 skill/resume/doctor 全部理解 team event log；
- 要么把 team 标 experimental/opt-in，先停止扩功能。

### P1-05：sync-public 仍是“可恢复但非事务”

`tools/sync-public.mjs:179-209` 仍先删公仓除 `.git` 外所有内容，再复制 export tree。若 `git add/diff/commit` 在覆写后抛错，catch 只给人工恢复命令，不自动 rollback。

此外：

- `:156-163` 只对自定义 token 脱敏；邮箱和用户目录仍原文打印；
- `:127-132` 还把本地 `excludePaths` 原样写进日志；
- tracked-default-publish 让任何漏加 exclude 的私有文档默认公开；
- public repo 是 mutable workspace，不是 disposable clone。

建议改成 versioned allowlist + disposable clone/export branch：

1. tracked allowlist 声明允许公开的 top-level/files；
2. 临时 clone 中 overlay、scan、gate、commit；
3. 全 refs/history 和 exact package tgz 同扫；
4. 成功后 fast-forward 公仓，失败直接丢临时 clone；
5. 所有命中只报类别、文件、序号/长度，不复录值或 private path。

### P1-06：closeout 仍无事务保证

`src/closeout.mjs:74-129` 的“先 mutate，后 gate”仍在。虽然有误收口 runbook，但命令失败时用户得到的是半完成工作树，不是原状态。

正确顺序：

- 在临时 tree/index 预演全部 status、目标路径、README、双门；
- 或记录 before image，失败自动恢复；
- `git add` 应在所有验证绿后执行；
- dry-run 应展示完整 projected diff，而不仅是步骤文本。

### P2-07：文档与实现漂移

- `README.md:170` 仍只写“三件套行数护栏”，没有 est-token、总量、foldable stage；
- `locales/zh.json` 仍说“每轮全量读这套”，与 skill 分层读冲突；
- Scrollery docs 仍宣称该 skill “零 token 税”，与 259 万历史归因账直接冲突；
- Scrollery CI 仍钉 registry alpha.2，所以本地源码功能与消费行为不一致。

## 6. Scrollery ROI 与工作流裁决

### 6.1 为什么仍有必要

当前 Scrollery：

- 24 个 active task，14 个 archived worklog；
- 14 个 closeout 共 75 个候选；21 个 no-promotion，54 个实际提升；
- no-promotion 占 28%，说明 closeout 不是无脑复制，而有真实筛选；
- UIUX 长线约 30 次续作、95,974-token trio，跨 session 记忆不是伪需求；
- 当前 live worktree 用新引擎跑 `check/index` 均绿；上一轮的 2 个新增缺 line 已被流程纠正。

这些证据足以支持：core memory + closeout + gate 继续留。

### 6.2 为什么当前全套不划算

Scrollery `.github/workflows/docs-governance.yml`：

- push 仅 `main`，但实际分支是 `dev`；
- 2 OS × 2 Node；
- 每格跑 check/index/baseline/selftest；
- 旧 `ci.yml` 仍跑五套旧 docs/closeout/index/skill 门。

结果：dev push 不跑新门，PR 却同时付新旧双税。消费仓不应替 engine 项目重复跑全量 selftest/跨平台矩阵；它只需单 OS/单 Node 的 exact-version `check/index`。跨平台是 worklog-kit 上游发布门责任。

另有 24 个 active 中约 7 个所有阶段均已完成却未 closeout。显式用户授权不能取消，但 doctor 可立即提示“ready for closeout”，不必等 14 天 stale；只提示，不自动迁移。

### 6.3 推荐能力 profile

| Profile | 内容 | Scrollery 默认 |
|---|---|---|
| `memory-lite` | 单文件 checkpoint：目标、当前、下一步、阻塞、决策、验证 | 小型跨 session 任务 |
| `memory-core` | trio + bounded resume + candidate closeout | 长任务默认 |
| `docs` | frontmatter/link/closeout gate | 开启 |
| `generated-status` | lines/status/generated index | 按需 |
| `team` | event stream、owner、collaborators | 默认关闭/experimental |
| `ci-paranoid` | 多 OS/Node、engine selftest | 仅 worklog-kit 自身 |

`strict/brownfield` 应继续表示债务执法策略，但不要冒充功能 profile；两组配置应正交。

## 8. 最省 token、又不降施工质量的方案

按收益/风险排序：

### 8.1 立即可做

1. **skill 渐进披露**：hot skill 只留触发、起步、干活、接续、收口授权；815-token closeout 步骤明确转读仓内 runbook。
2. **移除热读重复**：Scrollery AGENTS 长任务不再默认读完整 `docs/worklogs/README.md`；需要归档索引或收口时再读对应 section/runbook。
3. **doctor 摘要默认**：只出 totals + top 5；`--verbose` 展开，`--json` 给机器。
4. **停止固定 heredoc**：先退回 agent-native edit；产品命令完成后再统一走 stdin。
5. **ready-to-close 提示**：全部 phase 完成即提醒用户裁决，减少死 active task。

仅拆 closeout + 移除 worklogs README 热读，长任务静态税理论上可从升级后的 6,739 降约 3,214，接近减半，而且不删除任何任务事实。

### 8.2 产品化机械命令

优先顺序：

1. `worklog start <task> --mode lite|trio`
2. `worklog list [--active|--ready]`
3. `worklog resume <task> --compact|--full`
4. `worklog note <task> --kind progress|finding|decision --stdin`
5. `worklog checkpoint <task>`：lossless ledger → bounded hot view
6. `worklog next-id` 或 task-scoped/hash ID，避免为找全局 F-ID 让模型扫全仓

这些命令减少的不是文档质量，而是模型为了定位 section、复制模板、查 ID、整文件 Edit 所付的无效上下文。

### 8.3 不应采用

- 为过 4k/6k 线删除未消费证据；
- 把 acceptance criteria、用户裁决、失败边界折成一句；
- 降低 closeout candidate coverage 或把 `verified=yes` 改 warn；
- 让模型凭猜测跳过 findings；
- 用 summary 覆盖唯一 canonical 账本；
- 因 doctor 输出太吵而直接关 doctor，不修输出层。

## 9. 建议路线图

### 0–2 天：发布与安全止血

1. 冻结任何基于 alpha.2 当前 tag 的“可复现”表述；不要移动公开 tag。
2. 定位 registry alpha.2 的 exact source/tarball，记录三方差异。
3. 从 skill 撤回固定 heredoc 通用指引；修 findings 盲追加章节错误。
4. token-audit 增 `--since/--until`、event timestamp、version/skill hash/task bucket、shell write 识别和 fixtures。
5. sync-public 统一脱敏全部命中与 private paths。

### 1 周：把减税做成产品能力

1. 实现 `start/list/resume/note/checkpoint` 最小闭环。
2. closeout section 从 hot skill 拆出；Scrollery AGENTS 调整热读路由。
3. doctor 默认摘要、team events 护栏、ready-to-close、可选 ratchet。
4. closeout 与 sync-public 改事务/临时 clone 模型。
5. README 同步 doctor、token 口径和竞品定位。

### 下一发布：恢复供应链锚点

1. bump 新版本，不能复用 alpha.2；
2. 净化快照先落公仓；
3. 从公仓 exact commit pack；
4. 扫 exact tgz、全 refs/history、package allowlist；
5. 安装 tgz 做真 bin e2e；
6. publish exact tgz + provenance；
7. 校 registry shasum；
8. tag exact source commit，复 pack 对 shasum。

### 部署到 Scrollery 后：10–20 个任务 A/B

必须在新旧版本、同类型任务间比较：

- p50/p90 kit-attributed token；
- cold resume token；
- time-to-first-correct-action；
- 首次恢复后的错误动作；
- correction/rework 次数；
- 遗漏约束/错误重犯；
- gate catch 数与有效命中率；
- closeout 提升项后续复用率；
- 全阶段完成后 >3 天仍留 planning 的比例。

成功门槛建议：

- p50 每任务 token 至少下降 25%；
- p90 active hot view ≤6k；
- 错误、返工、遗漏不增加；
- finished-but-active >3 天比例 <10%；
- gate 继续捕获真实问题。

停止条件：同类任务 token 降幅不足 20%，或任一质量代理持续恶化，则停止 trio 默认触发，降为 memory-lite；若 docs gate 长期无有效命中，则以 DocGuard/Vale 或更小的 closeout-only gate 替代。

## 10. 最终建议

### 对 Scrollery

继续用，但先不升级宣称成功。正确顺序：

1. 修安全写入和可归因 audit；
2. 发布新版本并恢复 release/tag/tarball 一致性；
3. Scrollery upgrade；
4. 修 `dev` CI 和新旧双门；
5. 跑真实 A/B；
6. 只在质量代理不退化时扩大默认采用。

### 对 worklog-kit

停止横向加 feature，进入一轮“闭环与减法”：

- 核心：checkpoint/resume + closeout compiler + deterministic gate；
- 边缘：team/generated/paranoid opt-in；
- 度量：先可信，再优化；
- 发布：source/tag/tarball/provenance 一条链；
- 文案：不再卖“市场空白”，卖可验证的 engineering-memory lifecycle。

一句话结论：

> 本轮找对了最肥的长尾，却还没有把省 token 变成已部署、可度量、跨 agent、安全的产品能力；先补这条闭环，再谈扩张。

## 11. 局限

- token 估值使用 `o200k_base`；真实模型 tokenizer、cache 折扣和计费策略会不同。
- 反事实只模拟 progress 分层读，未假设未来 task_plan/findings 会按纪律继续缩小，因此对长期收益偏保守；同时没有计入错误恢复成本，因此不能替代 A/B。
- 竞品能力来自官方自述，未安装做同任务 benchmark。
- 本轮未修改产品代码、Scrollery workflow、公开仓、tag 或 registry；只新增复审文档与工作记忆。
- 安全结论不是法律或正式供应链审计意见；未复录任何本地 blocklist 真实值。
